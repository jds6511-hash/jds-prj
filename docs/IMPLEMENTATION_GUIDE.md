# 한눈에 보기 (요약)

*이 절만 읽어도 "지금 무엇을 만들면 되는지"를 알 수 있도록 정리한 요약이다. 세부 근거는 본문 해당 장을 참조한다.*

## 목표

업로드한 단일 영상(mp4)에서 자연어 문장 질의로 원하는 장면(세그먼트)을 찾는 검색 시스템을 만들고, **baseline(자막만 검색) 대비 proposed(자막+장면 캡션 결합 검색)의 성능 개선을 동일 질의셋으로 정량 검증**한다. 확장으로, 질의 없이 영상 전체를 훑어 사후검토(AAR) 스타일 리포트를 자동 생성하는 7번째 단계를 붙인다(제2부).

## 입력과 출력

- **검색 코어:** 입력 = mp4 + 자연어 질의 → 출력 = top-k 세그먼트 (정수 초 점프 위치 `[[시작,끝],...]` + 해당 자막)
- **AAR 확장:** 입력 = mp4 → 출력 = 모든 문장에 `[seg#N]` 근거 인용이 달린 사후검토 리포트

## 파이프라인 한 줄 요약

① 5초 고정 세그먼트 분할 → ② 세그먼트별 대표 프레임 1장 선택 → ③ 자막(Whisper large-v3) + 장면 캡션(Qwen2.5-VL) 생성·임베딩(KURE 또는 BGE-M3) → ④ per-query min-max 정규화 후 α 가중합 검색 → ⑤ 평가(Hit/Recall@k 주지표) → ⑥ 프로토타입(시점 점프 데모) → (+⑦ AAR 리포트 생성, 제2부)

## 핵심 확정 결정 (변경하려면 근거 필요)

- **5초 고정 분할** — EventFormer의 DiDeMo window 비교 결과 근거 (1장). 11주차에 3/5/10초 자체 ablation으로 재확인 (9-3).
- **VLM은 캡션 생성기로만 (frozen)** — 검색은 임베딩 코사인으로 분리. LMR·SG-VLM 선례 (7-2, 7-9).
- **자막·캡션·질의는 동일 임베딩 모델로 인코딩** — 통일된 의미공간 확보 (7-8).
- **연산 순서 고정:** s_sub·s_cap 계산 → 각각 min-max 정규화(단일 영상 범위) → 정적 세그먼트는 s_cap_norm 자리에 s_sub_norm 대입 → α 가중합. 순서를 어기면 α 실험이 왜곡된다 (8-2, 8-4).
- **α grid search는 dev셋에서만** (0.0~1.0, 0.1 간격) 수행 후 test에 고정 적용 — 데이터 누수 방지 (9-1).
- **주 평가지표 = 세그먼트 인덱스 기반 Hit/Recall@k, MRR.** IoU@0.5/0.3은 선행연구 비교용 보조 지표 (8-3).
- **클라우드 API(OpenAI 임베딩·Clova 등) 배제** — 온프레미스·오프라인 명분 유지.

## 이번 버전에서 할 것 / 안 할 것

**할 것:** 5초 분할, 대표 프레임 선택(SMART STC 차용), 자막·캡션 임베딩 인덱스, 정규화+가중합 검색, 질의 유형별(자막형/장면형/복합형) 평가, 시점 점프 데모, AAR 리포트 생성+이중 지표 평가(제2부).

**안 할 것 (향후과제로 명시):** 검색기 자체 학습(PREM 계열), 오디오 제3 모달리티(SMART), 이벤트 기반 가변 분할(EventFormer 학습부), scene graph 구조화 캡션(SG-VLM), MLLM fine-tuning, 학습 기반 융합(PREM-style learned fusion).

# 0. 큰 그림과 전제

본 가이드는 6편의 선행연구(TVR, PREM, EventFormer, Chrono, Video Enriched RAG, SMART)를 "구현 레시피"로 환원한 것이다. 단, 이 문서는 corpus-level VCMR보다 업로드한 단일 영상 안에서의 segment retrieval을 목표로 하며, 논문 근거와 프로젝트 가정을 분리해서 적는다. 핵심 전제 두 가지를 먼저 분명히 한다.

- 학습 vs 비학습: PREM·EventFormer·SMART는 모두 학습 기반(contrastive learning, adversarial training, LoRA fine-tuning)이라 14주 안에 그대로 재현하기 어렵다. 본 프로젝트의 임베딩 기반 의미검색 설계가 더 현실적이며, 아래는 "학습 없이 차용 가능한 설계 결정"만 골라낸 것이다.

- 범위 고정: 오디오를 제3 모달리티로 넣는 것(SMART), 검색기 자체를 학습시키는 것(PREM)은 본 프로젝트 범위를 넘는다. 향후과제로 둔다.

**전체 파이프라인은 다음 순서로 진행된다: ① 세그먼트 분할 → ② 대표 프레임 선택 → ③ 자막·캡션 생성 및 임베딩 → ④ 모달리티 결합 검색 → ⑤ 평가 → ⑥ 프로토타입. 문제 설정은 단일 업로드 영상에 대한 segment retrieval로 통일한다. (제2부에서 ⑦ AAR 리포트 생성이 추가된다.)**

## 0-1. 단계별 참조 논문 매핑

| --- | --- | --- |
| ① 세그먼트 분할 (1장) | EventFormer | 5초 고정(window) 정당화 + TSM 샷 경계 표시 |
| ② 대표 프레임 선택 (2장) | SMART (STC 1단계) | 프레임 차분 L2 norm으로 동적 프레임 선택 |
| ③ 자막·캡션 임베딩 (3장) | Video Enriched RAG | 타임스탬프 동기화 텍스트 블록 + 텍스트 임베딩 |

| ⑥ 프로토타입 출력 (6장) | Chrono, SMART | 정수 초 [[시작,끝],...] 중첩 리스트 형식 |

# 1. 세그먼트 분할 (파이프라인 1단계 · 전처리)

**무엇: mp4에서 오디오를 추출하고 5초 고정 세그먼트로 분할한다. 본 프로젝트의 기존 설계를 그대로 유지한다.**

왜: EventFormer는 세 가지 분할 전략(contrastive convolution, Kmeans, window)을 비교했는데, DiDeMo(5초 단위 분할 데이터셋)에서는 고정 window가 가장 우수했다. 즉 5초 고정 분할은 단순하면서도 근거 있는 선택이다.

선택적 개선 (여유 시): EventFormer의 TSM(Temporal Self-Similarity Matrix) 아이디어를 가볍게 빌려, 인접 프레임 임베딩 코사인 유사도가 급변하는 지점을 "샷 경계"로 표시해 둔다. 나중에 "어떤 질의가 샷 경계와 어긋나 검색 실패했나" 분석에 쓸 수 있다. 학습 불필요, numpy로 계산 가능.

**의사코드 — 오디오 추출 + 5초 세그먼트 분할:**

```
audio = ffmpeg_extract_audio(video, sr=16000, mono=True)   # wav/16kHz
duration = get_duration(video)
segments = []
for start in range(0, int(duration), 5):
    segments.append({'idx': len(segments),
                     'start': start,
                     'end': min(start + 5, duration)})

# (선택) 샷 경계 표시: 인접 프레임 임베딩 유사도 급락 지점
# sims[i] = cos(frame_emb[i], frame_emb[i-1])
# boundary = i where sims[i] < mean(sims) - delta
```

# 2. 대표 프레임 선택 (파이프라인 2단계)

**무엇: 각 5초 세그먼트에서 VLM 캡션을 생성할 대표 프레임 1장을 고른다.**

왜: 단순히 중간 프레임을 쓰는 대신, SMART의 키프레임 선택(STC 1단계)을 차용하면 "가장 동적인(정보량 많은)" 프레임을 골라 캡션 품질을 높일 수 있다. SMART는 인접 프레임 차분의 L2 norm을 구해 가우시안 평활화 후 값이 큰 프레임을 키프레임으로 선택한다. 이 부분은 학습이 전혀 필요 없다. (구현 확정: 차분은 픽셀 수로 정규화한 RMS — 해상도 독립, DESIGN_SPEC 4-2 참조.)

**의사코드 — 세그먼트 내 대표 프레임 선택 (SMART STC 1단계 차용):**

```
frames = sample_frames(segment, fps=3)          # 세그먼트 내 프레임들
#   (구현 확정: 세그먼트별 시크 대신 영상 1회 순차 디코딩으로 동일 프레임을 수집 —
#    m2_keyframe.sample_segments_sequential, 동등성 실측 검증. DESIGN_SPEC 4-2)
embs = [image_encoder(f) for f in frames]       # 또는 raw pixel diff도 가능
diffs = [l2_norm(embs[i] - embs[i-1]) for i in range(1, len(embs))]
diffs = gaussian_filter(diffs, sigma=1.0)       # 지터 억제
rep_idx = argmax(diffs) + 1                     # 가장 동적인 프레임
rep_frame = frames[rep_idx]

# 주의: 정적인 세그먼트는 중간 프레임으로 fallback
# 이때 is_static 플래그를 세그먼트에 기록 (4장·8-4에서 참조)
```

*주의: 화면 변화가 거의 없는 정적 세그먼트에서는 차분이 모두 작으므로, 그럴 땐 중간 프레임을 기본값으로 쓴다. 정적 판정 플래그(is_static)는 8-4의 캡션 환각 예외 처리에서 사용하므로 세그먼트 속성으로 반드시 기록한다.*

# 3. 자막·캡션 생성 및 임베딩 (파이프라인 3단계 · 핵심)

**무엇: 세그먼트마다 (a) Whisper로 자막을 만들고 (b) VLM으로 대표 프레임 캡션을 만든 뒤, 둘을 타임스탬프와 함께 하나의 텍스트 블록으로 정렬(align)하여 임베딩한다.**

왜: Video Enriched RAG가 본 프로젝트 설계의 청사진이다. 이 논문의 Table 3은 자막(ASR)이 시각 캡션 단독보다 검색 성능이 높고(HIT@1: 0.741 vs 0.65), 둘을 합친 정렬 transcript가 가장 우수함을 보였다. 이는 baseline(자막) vs proposed(자막+캡션) 설계의 정당성을 뒷받침한다.

정렬 텍스트 블록 형식 (각 5초 세그먼트 = 검색 단위 1개):

```
scene#7: "주방에 재료들이 나무 도마 위에 놓여 있다"   # VLM 캡션
audio_transcript: "재료를 미리 준비해 두세요"           # Whisper 자막
timestamp: 00:35                                        # 세그먼트 시작
```

이 블록 하나를 통째로 임베딩하면 자막+장면이 자연스럽게 결합된다. baseline은 audio_transcript 줄만, proposed는 블록 전체(또는 자막·캡션 임베딩을 4장에서 결합)를 쓴다.

**임베딩 모델: 원 논문은 OpenAI text-embedding-3-small을 썼으나, 본 프로젝트는 로컬 GPU 환경이므로 오픈소스 모델을 쓴다. 한국어 포함이면 KURE 1순위, BGE-M3 / multilingual-e5-large 대안 (7-8 참조).**

**의사코드 — 세그먼트별 인덱싱:**

```
for seg in segments:
    seg['subtitle'] = whisper_transcribe(seg, model='large-v3')
    seg['caption']  = vlm_caption(seg['rep_frame'])
    # 두 가지 인덱스를 따로 저장 (4장 결합을 위해)
    seg['emb_sub'] = embed(seg['subtitle'])     # baseline용
    seg['emb_cap'] = embed(seg['caption'])      # proposed용
    # (대안) 정렬 블록 통째 임베딩
    seg['emb_joint'] = embed(make_aligned_block(seg))

vector_index = build_index([s['emb_sub'] for s in segments],
                           [s['emb_cap'] for s in segments])
```

# 4. 모달리티 결합 검색 (파이프라인 4단계)

**무엇: 질의 문장을 임베딩해 자막 인덱스·캡션 인덱스에서 각각 유사도를 구하고, 가중합으로 최종 점수를 낸다. baseline은 자막만, proposed는 결합.**

왜: "어떻게 결합하느냐"가 7주차 설계의 핵심이다. PREM은 modality-specific 통찰(행동 묘사는 시각에, 대화는 자막에 더 관련)을 학습으로 구현했지만, 본 프로젝트는 학습 없이 가중치 α를 grid search로 정하면 된다. PREM의 λ=0.5는 초기 탐색값일 뿐이며, 본 프로젝트에서는 dev셋 grid search로 다시 확인한다.

**의사코드 — 가중치 결합 검색 (개념 골격):**

```
def search(query, alpha):
    q = embed(query)
    scores = []
    for seg in segments:
        s_sub = cos(q, seg['emb_sub'])
        s_cap = cos(q, seg['emb_cap'])
        score = alpha * s_sub + (1 - alpha) * s_cap   # proposed
        scores.append((seg['idx'], score, seg['start']))
    return sorted(scores, key=lambda x: -x[1])

# baseline = search(query, alpha=1.0)   # 자막만
# proposed = search(query, alpha=0.5)   # 결합 (α는 grid search로)
```

**실험화: 검증셋에서 α를 0.0~1.0으로 0.1 간격으로 바꿔가며 Recall@k가 최대가 되는 지점을 찾는다. 이 grid search 자체가 좋은 실험 결과 표("α별 성능")가 된다.**

*중요(확정 로직): 위 의사코드는 개념 골격이며, 실제 구현은 8-2·8-4에서 확정한 연산 순서를 따른다 — ① s_sub·s_cap 계산 → ② 각각 min-max 정규화(단일 영상 범위) → ③ 정적 세그먼트는 s_cap_norm 자리에 s_sub_norm 대입 → ④ 가중합. 정규화 없이 결합하면 α 실험이 왜곡되고, 정규화 후 정적 처리를 하지 않으면 점수 불연속성 버그가 생긴다. 확정 버전 의사코드는 8-4에 있다.*

# 5. 평가 (파이프라인 5단계)

**무엇: baseline vs proposed를 동일 질의셋으로 비교한다. 주지표는 세그먼트 인덱스 기반 Hit/Recall@k와 MRR, 보조지표는 IoU@0.5/0.3 (8-3에서 확정한 기준).**

왜: TVR과 모든 후속 논문이 Recall@k에 시간 IoU(Intersection over Union) 임계값(0.5, 0.7)을 함께 적용한다. 다만 본 프로젝트는 5초 고정 세그먼트라 정답 구간이 격자 경계를 가로지르면 IoU@0.5를 구조적으로 넘기 어려운 경우가 생기므로(상세: 8-3), 정답을 "정답 세그먼트 인덱스"로 라벨링해 segment-level Hit/Recall@k를 주지표로 삼고 IoU는 선행연구 비교용 보조지표로 병기한다. 정답 세그먼트와 질의는 영상 단위로만 분리해 누수를 막는다.

**의사코드 — 평가 지표:**

```
def iou(pred_seg, gt_seg):
    inter = max(0, min(pred.end, gt.end) - max(pred.start, gt.start))
    union = (pred.end - pred.start) + (gt.end - gt.start) - inter
    return inter / union

def hit_at_k(results, gt_seg_idx, k):        # 주지표 (8-3)
    return 1.0 if gt_seg_idx in [r.idx for r in results[:k]] else 0.0

def recall_at_k_iou(results, gt, k, thr=0.5):  # 보조지표
    return 1.0 if any(iou(r, gt) >= thr for r in results[:k]) else 0.0

def mrr(results, gt_seg_idx):
    for rank, r in enumerate(results, 1):
        if r.idx == gt_seg_idx:
            return 1.0 / rank
    return 0.0

# 전체 질의셋 평균 → 최종 Hit@1/5/10, MRR (주) + IoU@0.5/0.3 (보조)
```

**질의 유형별 분석: SMART·TVR처럼 질의를 "자막형 / 장면형"으로 나눠 각각의 성능을 따로 집계하면, "장면 결합이 어떤 질의에서 효과적인가"를 보여주는 핵심 분석이 된다(10주차).**

## 5-1. 평가셋 구성 원칙

질의는 자막형 / 장면형 / 복합형으로 태깅하고, 각 질의는 하나의 정답 세그먼트(start/end timestamp + 정답 세그먼트 인덱스)와 연결한다.

train/dev/test는 영상 단위로 나누고, α grid search는 dev에서만 수행한 뒤 test에 최종 적용한다. 분할 시 질의 유형(자막형/장면형/복합형) 비율이 세 split에서 가능한 한 유사하도록 층화를 고려한다 — dev의 유형 분포가 test와 다르면 α 선택이 편향된다.

평가 보고는 Hit/Recall@1/5/10, MRR을 주지표로 하고, IoU@0.5를 기본 보조지표, 필요하면 IoU@0.3도 병기한다 (8-3 확정 기준).

# 6. 프로토타입 출력 (파이프라인 6단계)

**무엇: 검색 결과를 시점 점프 + 관련 자막 표시로 보여주는 데모.**

왜: Chrono와 SMART가 공통으로 쓰는 출력 형식이 정수 초 단위 [[시작, 끝], ...] 중첩 리스트다. Chrono는 타임스탬프를 정수 초로 표현하는 것이 소수점보다 낫다고 보고했다(토크나이저 일관성). 프로토타입의 시점 점프도 이 형식으로 통일한다.

output = {

## 7-2. LMR (Context-Enhanced VMR with LLMs, arXiv:2405.12540) --- proposed 설계의 가장 가까운 학술적 근거

LMR(Large Language Model-guided Moment Retrieval)은 본 프로젝트의
"VLM/LLM 캡션을 검색에 활용"이라는 핵심 전략의 가장 가까운 학술적
선례다. 아래에 문제의식·구조·평가·차용점을 본 프로젝트 관점에서
정리한다.

### (1) 문제의식 --- 왜 LLM 캡션이 필요한가

기존 VMR 방법은 질의 문장이 상세한 환경·인물·행동 묘사를 담을 때,
외형이나 동작이 비슷한 다른 순간들과 정답 순간을 구별하지 못한다. 논문의
예시는 "신발 선반 앞에서 쾌활하게 말하는 여성"이라는 질의로, "말하는
여성"과 "선반"에 맞는 순간이 여럿일 때 기준 모델(QD-DETR)이 정답을 못
찾는다. 핵심 통찰은 LLM이 생성한 텍스트의 고수준 의미 정보가 이런 시각적
사각지대를 보완해, 시각적으로 비슷한 이벤트를 텍스트 관점에서 구별하도록
돕는다는 것이다. 이는 본 프로젝트가 "자막만으로는 못 찾는 무발화 장면을
장면 캡션으로 보완"하려는 가설과 동일한 동기다.

### (2) 구조 --- 3개 모듈

LMR은 세 모듈로 구성된다.

-   LLM 기반 시각 맥락 강화(VCE): 본 프로젝트와 가장 닮은 부분이다.
    논문은 MLLM이 영상·텍스트를 동시에 처리할 수 있지만 VMR에 직접
    적용하면 신뢰도가 떨어진다고 지적한다(MLLM이 시간 추론보다 지식 기반
    텍스트 생성에 최적화되어 있기 때문). 그래서 가중치를 고정한(frozen)
    MLLM의 텍스트 생성 능력만 활용해 각 클립의 설명을 만들고, 거기서
    배경·인물 외형 같은 대상 관련 맥락 정보를 추출한다. 즉 MLLM을
    검색기가 아니라 캡션 생성기로만 쓰는 것이며, 이는 본 프로젝트가
    VLM으로 장면 캡션을 뽑아 텍스트 검색에 투입하는 설계와 정확히 같은
    발상이다.
-   Video-Query Fusion(VQF): LLM 생성 설명에는 클립 간 중복 정보가
    많으므로, 교차 어텐션 층으로 질의 언어에 기반해 관련 맥락만
    부각시킨다. 중요한 설계 디테일은, 질의 특징과 텍스트(캡션) 특징을
    같은 텍스트 인코더로 추출해 통일된 의미 공간에 둔다는 점이다. 본
    프로젝트가 자막·캡션·질의를 같은 임베딩 모델로 인코딩하는 원칙과
    일치한다.
-   언어 조건부 트랜스포머: 자유 형식 언어 질의를 정렬된 영상 표현으로
    즉석에서 디코딩해 순간을 찾는다. 시각 백본은 CLIP과 SlowFast, 텍스트
    백본은 CLIP-Text를 사용한다.

### (3) 캡션 생성 디테일 --- 차용 가치 높음

LMR은 BLIP-2와 StableVicuna 기반의 VideoChat을 맥락 생성기로 사용하고,
ChatGPT로 만든 지시문(instruction) 목록(예: "다음 영상을 간결하게
설명하라" 등 10개 변형)에서 각 클립마다 무작위로 하나를 골라 다양한
스타일의 설명을 생성한다. 상세형("주황색 블라우스를 입은 여성이 카메라
앞에서 하늘색 셔츠의 남성과 대화한다")부터 간결형("전화 통화하는
사람")까지 폭이 넓다. 생성된 설명을 시간 순서로 정렬해 영상 시퀀스의
시간적 맥락을 구성한다. 이 "다중 지시문(multi-view) 캡션 + 타임스탬프
정렬"은 본 프로젝트의 3단계(세그먼트별 VLM 캡션 → 타임스탬프 정렬 →
임베딩)와 개념적으로 동일하며, 프롬프트를 한 종류로 고정하지 말고
간결형/상세형 등 몇 가지로 변형해 캡션 다양성을 높이는 11주차 고도화
실험으로 응용할 수 있다.

### (4) 평가·성능

LMR은 QVHighlights·Charades-STA 벤치마크에서 최근접 경쟁모델 대비 각각
최대 3.28%·4.06% 향상을 보고했고, 특히 복잡한 질의(긴 문장·다중 절)의
위치추정에서 이득이 두드러졌다. 이를 위해 기존 검증셋에서
절(clause)·단어 수 기준으로 복잡 질의만 모은 C-QVal split을 따로 구성해
복잡 질의 효과를 별도 측정했다. 본 프로젝트도 질의를
자막형/장면형/복합형으로 태깅하므로(5-1 참조), "복합형 질의에서
proposed의 이득이 더 큰가"를 LMR의 C-QVal 발상처럼 별도 집계하면 강한
분석이 된다.

본 프로젝트에 가장 결정적인 부분은 ablation 결과다. 영상 기반 LLM 대신
이미지 기반 LLM(LLaVA)으로 교체하면 성능이 1.04~3.87% 감소하고, 일반
이미지 캡셔닝(ImgCap)으로 바꾸면 3.3~4.66%로 더 크게 떨어진다. 또한 두
VQF 모듈의 가중치 공유(weight sharing)를 제거하면 R1@0.5가 약 2.13%
하락한다. 즉 이미지 기반 LLM이 시간·오디오 정보가 없어도, 웹 스케일
사전학습 덕분에 일반 캡셔너보다 더 유효하고 상세한 장면 정보를 제공한다.
이는 "왜 일반 캡셔너가 아니라 VLM/LLM 캡션을 써야 하는가"에 대한 직접
근거이며, 본 프로젝트가 3단계에서 VLM 캡션을 채택한 설계를 뒷받침한다.

### (5) 본 프로젝트와의 정렬·차이

가장 강한 정렬점은 "MLLM/VLM을 캡션 생성기로만 쓰고 그 텍스트를 검색에
활용한다"는 핵심 전략이다.

차이는 두 가지다. 첫째, LMR은 VQF·언어 조건부 트랜스포머를 학습시키는
학습 기반 모델이지만, 본 프로젝트는 학습 없이 사전학습 임베딩의 코사인
유사도로 검색한다(14주 일정에 현실적). 둘째, LMR은 LLM 캡션을 시각
특징과 결합(concatenate)해 영상 표현을 강화하는 반면, 본 프로젝트는
캡션을 독립 텍스트 인덱스로 두고 자막 인덱스와 가중합(alpha*s_sub +
(1-alpha)*s_cap)한다.

결과 보고서 선행연구 절에서는 "원시 영상 특징을 강화하는 학습 기반(LMR)
vs VLM 캡션 텍스트를 학습 없이 결합(본 연구)" 대비로 위치를 명확히 하면
좋다.

(전문 확인으로 추가된 디테일) LMR은 LLM(VideoChat)을 전처리 단계에서
오프라인으로 단 한 번만 돌려 클립 설명을 생성하고, 이후 학습·추론에는
미리 추출한 특징만 쓴다(단일 RTX 3090에서 학습 2~5시간, 추론 영상당
0.45초). 이 "LLM은 인덱싱 때 1회만, 검색은 임베딩만"이라는 구조가 본
프로젝트와 정확히 일치하며, 본 프로젝트의 "VLM은 캡션 생성기로만 분리"
설계를 강하게 정당화한다. 단, 주의: LMR의 가벼운 비용은 사전 추출 특징을
쓰기 때문이며, 본 프로젝트는 VLM 캡션 생성(추론) 자체를 직접 수행하므로
병목은 학습이 아니라 캡션 생성 시간이 된다. 또한 LMR ablation에서 영상
특징을 제거하고 캡션만 쓰면(w/o VFE) 성능이 크게 떨어졌는데(R1@0.5
65.48→55.13), 이는 "캡션 단독은 약하고 결합이 핵심"임을 보여주며 본
프로젝트의 alpha grid search(자막·장면 가중치 탐색)가 왜 중요한지를
뒷받침한다.

*주의: 위 수치(1.04~4.66%, 2.13%, 3.28%·4.06%)는 LMR이
QVHighlights·Charades-STA에서 자체 학습 모델로 보고한 값이다. 본
프로젝트의 vlog 데이터·5초 세그먼트·학습 비의존 임베딩 환경과 직접
비교할 수 없으므로 "선행연구가 보고한 경향"으로만 인용한다.*

## 7-3. Modal-Enhanced Semantic Modeling --- 질의 균형 설계 근거

이 연구는 VMR에서 드물게 모달리티 불균형 문제를 정면으로 다룬다. 생성
캡션으로 서포트셋을 구축하는 선행 연구를 언급하면서, 같은 영상 내
세그먼트 문장들 간의 인과 관계를 무시한 채 단순히 네거티브로 취급하는
기존 방식의 한계를 지적한다. 본 프로젝트가 4주차에서 자막형/장면형
질의를 균형 있게 구성하려는 설계, 그리고 같은 영상 내 세그먼트를 누수
없이 다루려는 평가 원칙(5-1)과 직접 맞닿는다.

## 7-4. SDS KoPub VDR --- 한국어 "텍스트 vs 멀티모달" 비교 틀

한국어 공공문서 시각 문서 검색 벤치마크로, 텍스트 임베딩 모델·멀티모달
임베딩 모델·자체 모델 세 범주를 비교해 텍스트 기반 검색과 멀티모달
임베딩 접근의 상대적 강점·한계를 정량 분석한다. 이는 본 프로젝트의
"자막만(baseline) vs 자막+장면(proposed)" 비교 실험과 동일한 골격이라,
한국어 환경에서의 비교 방법론·지표 설계 선례로 인용할 수 있다.

## 7-5. KURE --- 한국어 임베딩 모델 보강 옵션

기존 가이드는 BGE-M3 / multilingual-e5-large를 권장했는데, 한국어 특화
옵션으로 KURE를 추가 검토한다. KURE는 기존 다국어 임베딩이 한국어 고유
특성을 온전히 반영하지 못하고 장문 검색에서 성능이 떨어지는 한계를
겨냥한 한국어 특화 검색 임베딩으로, 한국어 검색 벤치마크에서 강력한
다국어·상용 모델을 상회하며 특히 장문 검색에서 경쟁력을 보고했다. 한국어
vlog 자막·캡션을 임베딩한다면 BGE-M3 대비 후보로 비교해볼 가치가 있다.
(참고: 실서비스 BibiGPT 내부 테스트에서도 "프레임+자막" 결합 검색이 자막
단독 대비 Top-3 recall을 약 35% 올렸다고 보고되었으나, 이는 학술 출처가
아니므로 보조 정황 근거로만 사용한다.)

## 7-6. 보강된 환경·도구 권장 (업데이트)

-   임베딩(한국어 우선): KURE(한국어 특화, 장문 강점) 또는 BGE-M3 /
    multilingual-e5-large 중 dev셋에서 비교 선택
-   비교 방법론 참조: SDS KoPub VDR(한국어 텍스트 vs 멀티모달 비교 틀)

## 7-9. frozen VLM 활용 추가 선례 및 구조화 캡션 (SG-VLM, arXiv:2509.11862)

성격: 본 프로젝트의 직접 채택 대상이 아니라, (a) frozen VLM 활용 전략의
추가 선례, (b) "구조화 캡션"이라는 향후과제 방향, (c) "장면 결합이 항상
이득은 아니다"라는 한계의 근거로 인용한다.

개요. SG-VLM(플로리다대, Haodi Ma 외)은 영상 질의응답(VideoQA)에서 VLM이
얕은 상관관계에 의존해 시간·인과 추론이 약하고 환각을 일으키는 문제를
다룬다. 해법은 VLM이 정답을 바로 내놓는 대신, 중간에 symbolic scene
graph(객체를 노드, 관계를 엣지로 표현한 구조)를 먼저 생성하고 이를
근거로 답하게 하는 것이다. 핵심은 별도 학습된 무거운 scene graph 모델
없이, frozen VLM을 프롬프팅만으로 scene graph를 생성한다는 점이다.

3단계 구조: ① Scene Graph 생성(프레임별 객체·관계 추출, 예: (주황
고양이, 보고 있다, 먹는 얼룩 고양이)), ② Scene Graph 선택(질문과 관련된
그래프만 남기는 query-aware filtering), ③ grounded 답변 생성(선택된
그래프 + 영상 맥락으로 VLM 프롬프팅).

사용 모델(검증됨): 백본은 Qwen2.5-VL과 InternVL. baseline에
BLIP-2·Flamingo 포함. 단, 공간 관계 추출에는 GroundingDINO(바운딩
박스) + Segment Anything(정제) +
Metric3Dv2·WildCamera·PerspectiveFields(3D 좌표 투영)를 줄줄이 사용한다
--- scene graph 생성이 VLM 하나로 끝나지 않고 객체검출·분할·깊이추정
모델 스택을 요구한다는 의미.

핵심 결과·한계: 세 벤치마크(NExT-QA, iVQA, ActivityNet-QA)에서 인과·시간
추론을 개선하고 baseline을 능가했으나, 강력한 VLM 대비 이득은
제한적이었다. 질문 관련 프레임만 쓰는 FrameSel-SG가 전체
사용(Full-SG)이나 VLM 단독보다 일관되게 우수했는데, 무관한 프레임이
distracting 객체·관계로 추론을 희석시키기 때문이다. 논문은 부정확한
scene graph가 오히려 성능을 떨어뜨릴 수 있음을 솔직히 인정한다.

본 프로젝트와의 관계:

-   차용: frozen VLM 프롬프팅 철학(학습 없이 VLM 활용) --- LMR에 이은
    추가 선례. Qwen2.5-VL 백본 사용은 본 프로젝트의 캡션 모델
    선택(7-8)을 재차 뒷받침. query-aware selection 발상은 본 프로젝트가
    임베딩 유사도 랭킹으로 이미 부분 반영.
-   비채택: scene graph 생성 파이프라인(GroundingDINO+SAM+3D 깊이
    모델)은 14주 일정·단순 캡션 임베딩 설계에 비해 과도하게 무겁고, 비용
    대비 효과가 불확실. VideoQA 과제라 검색(retrieval)인 본 프로젝트와
    과제 자체가 다름.
-   한계의 활용: "장면 정보 결합이 항상 이득은 아니며 캡션·구조 품질에
    의존한다"는 점은, 본 프로젝트의 baseline(자막) vs
    proposed(자막+장면) 비교에서 proposed가 오히려 나빠질 수 있는 경우를
    미리 설명하는 근거. 13주차 보고서 한계·향후과제 절에 활용.
-   능동적 신규성 서사(권장): 단순히 "나빠질 수 있음을 안다"는 수동적
    방어를 넘어, 다음과 같이 능동적으로 서술한다 --- "SG-VLM(2025) 등
    최신 연구도 무거운 구조화 캡션(scene graph)이 비용 대비 이득이
    제한적임을 보였다. 이에 본 연구는 무거운 구조화 대신 가벼운 frozen
    VLM 단일 프레임 캡션(Qwen2.5-VL)에 8-4단계의 정적 구간 예외 처리
    필터를 결합하여, 환각 노이즈를 공학적으로 제어하는 실용적 결합
    방법론을 제안한다." 이 서사는 9-2의 신규성(제약 환경에서의
    실증적·공학적 기여)과 직접 연결된다.

*주의: 본 정리는 원 논문(arXiv:2509.11862) 및 다수 2차 리뷰로 교차
확인했으나, 세부 수치는 인용 시 원문 표를 재확인할 것.*

# 8. 구현·평가 시 보완사항 (리뷰 반영)

기존 설계(5초 고정 세그먼트, 가중합 검색, IoU 평가)에 대한 외부 리뷰를
검증해 채택한 보완사항이다. 모두 비용 대비 효과가 크고 시행착오를 줄여
준다. 우선순위가 가장 높은 것은 8-2(점수 정규화)다.

## 8-1. 자막-장면 동기화: 경계 걸친 발화의 오버랩 처리 (1·3단계)

문제: 5초 고정 분할에서 발화가 세그먼트 경계를 가로지르면(예: 4초에
시작해 8초에 끝남) 자막이 두 세그먼트로 쪼개져 문맥이 깨지고 임베딩
매칭이 저하된다.

보완: 5초 윈도우는 유지하되, 자막 할당 시 발화(문장) 단위를 깨지 않도록
하고, 인접 세그먼트와 1~2초 오버랩을 두어 경계 걸친 발화가 양쪽에 모두
포함되게 한다. Whisper의 단어/구 타임스탬프를 기준으로 "발화가 더 많이
걸친 세그먼트에 귀속 + 경계 문장은 양쪽 중복 허용" 규칙을 3단계 자막
정제에 명시한다. 비용 거의 없음.

## 8-2. 모달리티 점수 스케일 정규화 (4단계) --- 최우선

문제: score = alpha*s_sub + (1-alpha)*s_cap 선형 결합에서, 자막
유사도(구어체 자막 분포)와 캡션 유사도(VLM 생성 캡션 분포)는 코사인 값의
분포·밀도가 다르다. 예컨대 자막이 0.6~0.8, 캡션이 0.4~0.6에 몰리면
alpha=0.5여도 자막 점수가 랭킹을 지배(dominate)해 공정한 grid search가
불가능하다.

보완: grid search 이전에 두 모달리티 유사도를 각각 정규화(min-max 또는
z-score)해 같은 스케일로 맞춘 뒤 가중합한다. 정규화 범위는 query별 후보
점수 집합 단위로 수행하는 것이 랭킹에 안전하다(corpus 전체 정규화는
query 간 분포 차이를 못 잡음). 이 보완 없이는 alpha 실험 결과 자체가
왜곡되므로 반드시 반영한다.

정규화 범위(아키텍처 정책): 정규화 스케일링은 반드시 "해당 단일 영상 내
세그먼트 집합"으로 엄격히 제한한다. 단일 DB에 여러 영상이 묶이는 멀티
영상 환경에서도, 하나의 query에 대해 그 query가 대상으로 하는 영상
하나의 세그먼트 유사도 배열에 대해서만 min-max를 계산한다(영상 경계를
넘는 정규화 금지). 이 정책은 6단계 프로토타입 백엔드에 명시한다. 참고로
단일 영상은 세그먼트 수가 크지 않아(예: 1시간=720개) numpy 벡터 연산으로
즉석 정규화해도 연산 부담은 미미하다.

수정 의사코드(4단계):

    def search(query, alpha):
        q = embed(query)
        s_sub = [cos(q, seg['emb_sub']) for seg in segments]
        s_cap = [cos(q, seg['emb_cap']) for seg in segments]
        s_sub = minmax_normalize(s_sub)   # query별 후보 점수 정규화
        s_cap = minmax_normalize(s_cap)
        scores = [(seg['idx'], alpha*a + (1-alpha)*b, seg['start'])
                  for seg, a, b in zip(segments, s_sub, s_cap)]
        return sorted(scores, key=lambda x: -x[1])

## 8-3. 평가 기준: 세그먼트 인덱스 라벨링 기반 Hit/Recall@k 주지표화 (5단계)

문제: 5초 고정 세그먼트는 단일 구간을 예측하므로, 정답 구간이 격자
경계를 가로지르면(예: 정답 3~7초) 어떤 단일 5초 세그먼트도 IoU 0.5를
구조적으로 넘기 어렵다(0~5초·5~10초 모두 교집합 2초). 핵심 장면을
맞췄는데도 IoU@0.5 기준으로는 실패 처리되는 억울한 경우가 생긴다.

보완(처방 정정): 단순히 IoU 임계값을 0.3으로 낮추는 것보다, 정답을 "정답
세그먼트 인덱스"로 라벨링해 평가를 segment-level Hit/Recall@k로
단순화하는 것이 5초 고정 구조에 더 정확하다. 즉 정답 구간과 가장 많이
겹치는 세그먼트(들)를 정답 인덱스로 지정하고, "top-k 안에 정답
세그먼트가 포함되는가(Hit)"를 주 지표로 삼는다. IoU 기반
지표(IoU@0.5/0.3)는 선행연구 비교용 보조 지표로 병기한다. 본 프로젝트
데이터 명세서가 이미 정답 타임스탬프를 라벨링하므로, 거기에 "정답
세그먼트 인덱스" 칼럼을 추가하면 추가 비용 없이 적용된다.

## 8-4. 정적 세그먼트 캡션 환각 예외 처리 (2단계)

문제: 화면 변화가 거의 없는 정적 세그먼트(카메라 고정, 음성만)에서 중간
프레임 1장만 VLM에 주면, VLM이 화자 의도와 무관한 소품 위주의 캡션을
생성(hallucination)해 캡션 인덱스에 노이즈가 들어간다.

보완: 프레임 차분 L2 norm 평균이 임계값 이하인 완전 정적 구간은, 캡션을
버리는 대신 자막에만 의존하도록 처리한다. 단, 8-2에서 모달리티별
정규화를 도입했으므로 단순히 "alpha=1.0을 곱한다"고 하면 점수
불연속성(discontinuity) 버그가 생긴다.

불연속성 문제: 정규화 후 s_sub_norm·s_cap_norm의 상한이 각각 1.0인
상태에서, 정적 세그먼트에 score=1.0*s_sub_norm을 주면, 동적 세그먼트의
score=0.5*s_sub_norm+0.5*s_cap_norm과 스케일이 구조적으로 어긋난다.
캡션 점수가 낮은 동적 세그먼트는 상한이 1.0보다 작아지는데, 정적
세그먼트는 자막만으로 1.0에 도달할 수 있어 정적 세그먼트가 부당하게
유리해진다.

올바른 처방: 정적 세그먼트는 s_cap_norm 자리에 s_sub_norm을 대입한다.
그러면 score = alpha*s_sub_norm + (1-alpha)*s_sub_norm = s_sub_norm이
되어, "캡션 대신 자막을 한 번 더 본다"는 의미가 되고 스케일(상한 1.0)이
동적 세그먼트와 정확히 일치해 불연속이 사라진다. (s_cap을 최솟값으로
치환하면 정적 세그먼트가 부당하게 불리해지므로 쓰지 않는다.)

연산 순서 고정(4단계): 순서를 어기면 정규화 분포가 왜곡되므로 다음
순서를 반드시 지킨다. 1. 전체 세그먼트에 대해 s_sub, s_cap 계산 2.
s_sub, s_cap 각각 독립적으로 min-max 정규화 (해당 단일 영상 세그먼트
집합 범위) 3. 정적 세그먼트는 s_cap_norm 자리에 s_sub_norm 대입 (정규화
이후에 대입 --- 정규화 전에 치환하면 정적 세그먼트의 원본 s_cap이
min/max 계산을 오염시킴) 4. 최종 가중합 alpha 적용

수정 의사코드(2·4단계 연계):

    # 1) 계산
    s_sub = [cos(q, seg['emb_sub']) for seg in segments]
    s_cap = [cos(q, seg['emb_cap']) for seg in segments]
    # 2) 각각 정규화 (단일 영상 범위)
    s_sub_n = minmax_normalize(s_sub)
    s_cap_n = minmax_normalize(s_cap)
    # 3) 정적 세그먼트는 s_cap_norm 자리에 s_sub_norm 대입
    for i, seg in enumerate(segments):
        if seg['is_static']:            # 프레임 차분 평균 < threshold
            s_cap_n[i] = s_sub_n[i]
    # 4) 가중합
    scores = [(seg['idx'], alpha*a + (1-alpha)*b, seg['start'])
              for seg, a, b in zip(segments, s_sub_n, s_cap_n)]
    scores.sort(key=lambda x: -x[1])

2단계 대표 프레임 선택의 fallback 규칙에 "정적 판정 플래그(is_static)"를
세그먼트 속성으로 기록해 4단계에서 참조한다. (비평 반영) 이 예외 처리가
proposed의 장면 기여를 깎거나 정적/동적 세그먼트 비율에 따라 비교
공정성을 왜곡하는지 확인하기 위해, 11주차에 예외 처리 on/off ablation을
둔다. 아울러 baseline은 동일 파이프라인에서 α=1.0인 특수 경우이므로,
정규화·예외 처리 인프라가 proposed와 동일해 비교가 대칭적이라는 점을
보고서에 명시한다.

## 8-5. 한국어 임베딩: 장면형 질의에서 KURE vs BGE-M3 분리 검증 (7-8 연계)

문제: KURE 같은 retrieval 임베딩은 보통 [짧은 질의 → 긴 문서] 비대칭
매칭에 최적화된다. 자막(구어체 장문)에는 잘 맞겠지만, 질의(짧은 문장) →
캡션(짧은/중간 문장)의 대칭(문장-문장) 매칭에서도 BGE-M3보다 우수한지는
별개 문제다.

보완: 7~8주차 dev셋 임베딩 비교에서 전체 성능만 보지 말고, 질의
유형별로 분리해 평가한다. 특히 장면형 질의(시각 중심, 캡션과 매칭되는
케이스) 부분집합에서 KURE와 BGE-M3의 Recall@k를 따로 집계해 대칭 매칭
변별력을 확인한다. 본 프로젝트는 이미 질의를 자막형/장면형으로
태깅하므로 추가 비용이 없고, 모델 선택 근거가 보고서에서 훨씬 설득력
있어진다.

요약: 8-2(점수 정규화)는 실험 타당성의 전제이므로 필수. 8-1·8-4(전처리
디테일), 8-3(평가 기준 정정), 8-5(임베딩 분리 검증)는 비용 대비 효과가
커 모두 채택 권장.

# 9. 예상 비판과 방어 논리

외부 적대적 비평(60개 후보 제기, 전부 반박·기각, 확정 약점 0건)에서
반복적으로 나온 주제 3가지에 대한 선제 방어 논리다. 발표·심사에서 재차
나올 수 있으므로 문서에 명시해 둔다.

## 9-1. 재현성 (세부 고정값)

 "라벨링 기준·grid search 범위·IoU 계산
단위·출력 규칙이 고정되지 않았다"는 지적에 대해, 다음을 고정값으로
못박는다. (a) alpha grid search는 0.0~1.0, 0.1 간격, dev셋에서만 수행,
동률 시 alpha가 큰 값(자막 우선) 선택. (b) 평가는 segment-level
Hit/Recall@k 주지표(8-3), IoU@0.5/0.3은 보조. (c) 질의 태깅은
자막형/장면형/복합형 가이드라인을 문서화하고 2인 이상 라벨링 후 불일치는
협의 조정하며, 조정 전 라벨러 간 일치율을 보고서에 병기한다. (d) 출력은
정수 초, 세그먼트 시작은 내림·끝은 min(start+5, duration). 이 고정값들은
데이터 명세서에 기록한다.

## 9-2. 신규성 (포지셔닝)

 "5초 분할·정렬 블록·가중합·평가지표가 모두
선행연구 차용이라 독자적 신규성이 약하다"는 지적에 대해, 본 프로젝트의
기여는 새 알고리즘이 아님을 명확히 한다. 기여는 (1) 한국어 vlog·단일
업로드 영상·비학습 임베딩이라는 실제 제약 환경에서, (2) 자막
단독(baseline) 대비 자막+장면 캡션 결합(proposed)의 효용을 동일 조건으로
정량 검증하고, (3) 어떤 질의 유형에서 결합이 이득/손해인지를 분리
분석하는 데 있다. 즉 방법의 신규성이 아니라 "검증된 요소들을 특정 제약
환경에서 통합·비교 검증"하는 실증적 기여로 위치를 잡는다. 나아가
SG-VLM(2025) 등이 밝힌 "무거운 구조화 캡션의 제한적 이득"을 근거로, 본
연구는 가벼운 frozen VLM 단일 프레임 캡션에 정적 구간 예외 처리(8-4)를
결합해 환각 노이즈를 공학적으로 제어하는 실용적 결합을 제안한다는 점에서
공학적 기여도 갖는다(7-9 능동적 서사 참조).

## 9-3. 5초 고정 분할의 전이 가능성

 "EventFormer의 DiDeMo 비교 결과가 단일
영상 segment retrieval에 그대로 전이된다는 보장이 없다"는 지적에 대해,
5초 고정은 근거 있는 기본값이되 절대적 선택이 아님을 인정한다. 11주차
고도화에서 세그먼트 길이(예: 3초/5초/10초)를 dev셋에서 비교하는 소규모
ablation을 두어, 본 프로젝트 데이터에서의 적정 길이를 자체 검증한다.
이로써 "차용한 가정을 프로젝트 환경에서 재확인"하는 절차를 명시적으로
갖춘다.

*비고: 위 세 가지는 확정된 결함이 아니라, 자동 비평에서 반복 등장한 예상
질문에 대한 사전 대비다. 대부분은 7-2·8-2·8-3·8-4 등에서 이미 다룬
내용을 심사 대응용으로 요약한 것이다.*

# 13. 프롬프트 설계

## 13-1. 구조화 출력 형식 --- `[seg#N]` 인용 강제

**원칙: 리포트의 모든 문장은 최소 하나의** `[seg#N]` **인용을 포함해야
한다.** 인용이 없는 문장은 14장의 Groundedness 계산 시 자동으로 "미인용
= 검증 불가"로 처리되어 감점된다. V2Xum-LLM의 프레임 인덱스 정렬 방식을
세그먼트 인덱스로 그대로 옮긴 것이다.

의사코드(프롬프트 골격):

    system: 당신은 영상 사후검토(AAR) 리포트 작성자입니다.
    아래는 5초 단위 세그먼트별 자막(subtitle)과 장면 캡션(caption)입니다.
    규칙:
    1. 모든 문장은 반드시 하나 이상의 [seg#N] 인용을 포함할 것.
    2. 세그먼트에 없는 내용은 절대 추측해 쓰지 말 것. 근거가 없으면 문장 자체를 생략할 것.
    3. 시간 순서대로 사건을 서술할 것.
    4. 인용한 seg#의 내용과 문장이 실제로 일치해야 함 (사후 검증됨).

    입력:
    [seg#0] 00:00-00:05 subtitle: "..." caption: "..."
    [seg#1] 00:05-00:10 subtitle: "..." caption: "..."
    ...

    출력 형식:
    - 사건 서술 문장 [seg#N]
    - 사건 서술 문장 [seg#N, seg#M]

## 13-2. Map-Reduce (긴 영상 대응)

세그먼트 수가 많아 전체를 한 번에 LLM 컨텍스트에 넣을 수 없는 긴
영상(예: 1시간 = 세그먼트 720개)에 대한 처리 절차.
Video-ChatGPT·MovieChat이 다루는 "긴 영상 압축" 문제의식을 검색이 아닌
생성 단계에 적용한 것이다.

-   **Map 단계:** 세그먼트를 N개(예: 60개 = 5분 분량)씩 묶어 청크별 부분
    리포트를 생성한다. 각 부분 리포트도 13-1의 `[seg#N]` 인용 규칙을
    동일하게 적용한다. 청크 경계에 걸친 사건이 쪼개져 누락되지 않도록
    인접 청크 간 5~10개 세그먼트의 오버랩을 둔다.
-   **Reduce 단계:** 청크별 부분 리포트들을 다시 하나의 LLM 호출로
    통합해 최종 리포트를 만든다. 이 단계에서는 (a) 중복 사건 제거, (b)
    시간 순서 재정렬만 수행하고 **새로운 사실을 추가하지 않도록**
    프롬프트에 명시한다 --- reduce 단계에서 새 내용이 생기면 map
    단계에서 인용했던 seg# 근거와 분리되어 환각 위험이 커진다.
    (안전장치) reduce 출력의 [seg#N] 인용 집합이 map 출력 인용 집합의
    부분집합인지 코드로 검사해, 통합 과정의 인용 유실·오귀속을 LLM 없이
    걸러낸다.

*주의: map-reduce 자체는 두 논문의 직접 재현이 아니라 "긴 컨텍스트 분할
처리가 필요하다"는 문제의식만 차용한 것임을 보고서에 명시한다(7-9절과
같은 방식의 차용 범위 명확화).*

# 14. 평가지표 설계

기존 4주차 평가 질의셋(질의-정답구간, 자막형/장면형/복합형 태깅)을
그대로 재활용하는 두 지표를 채택한다. 두 지표는 상충 관계다 --- 근거
없으면 생략하는 보수적 생성일수록 Groundedness는 오르고 Coverage는
내려가는 precision/recall 관계이므로, 한쪽만 최적화하지 않도록 반드시
쌍으로 보고한다. (선택, 여유 시) 정확성 지표가 재지 못하는 리포트
효용(가독성·구조·유용성)은 소규모 사람 평가로 보완할 수 있다.

## 14-1. Event Coverage Rate

**정의:** 정답구간과 연결된 각 세그먼트에 대해 "생성된 리포트가 이
세그먼트의 내용을 언급했는가"를 LLM-judge로 이진 판정한 뒤, 커버된
세그먼트의 비율을 계산한다. 분모는 전체 영상 세그먼트가 아니라 질의셋
정답구간과 연결된 세그먼트 집합이며, 이 한정을 결과 해석 시 명시한다.

**학술적 대응:** MiRAGE(arXiv:2510.24870)의 InfoF1(정보 커버리지·사실성)
지표와 구조적으로 동일하다. MiRAGE는 멀티모달 RAG 평가에서 claim 단위로
소스 대비 커버리지·사실성을 재는 프레임워크를 제안했는데, 본 지표는 이를
단일 영상 세그먼트 도메인에 적용한 것으로 위치를 잡는다.

**분리 집계:** 기존 질의 태깅(자막형/장면형/복합형)별로 나눠 집계하면,
"장면형 사건이 리포트에 얼마나 반영되는가"라는 8-5절과 대칭되는 분석이
자연스럽게 나온다.

## 14-2. Groundedness (환각률)

**정의:** 리포트의 각 문장이 인용한 `[seg#N]`의 실제 자막·캡션 내용과
일치하는지 LLM-judge로 검증한다. 복수 인용 문장([seg#N, seg#M])은
인용된 세그먼트 전부를 judge 입력에 함께 제공해 판정한다 --- 개별
세그먼트와만 대조하면 여러 세그먼트를 종합한 정당한 서술이 환각으로
오분류된다. 불일치하거나 인용이 없는 문장의 비율을 환각률로 계산한다.

**학술적 대응:** MiRAGE의 CiteF1(인용 근거 지지도·완전성)과
RAGAS(arXiv:2309.15217)의 Faithfulness 지표 모두와 대응된다. 특히
RAGAS는 정답 라벨 없이 Faithfulness·Answer Relevancy·Context
Precision·Context Recall 네 지표로 RAG 시스템을 평가하는 실무 표준
프레임워크라, "왜 이 방식이 검증됐는가"의 근거로 인용하기 좋다.

## 14-3. (선택, 여유 시) FActScore 스타일 세분화 검증

FActScore(Min et al., arXiv:2305.14251)는 생성 텍스트를 원자적
사실(atomic fact) 단위로 쪼갠 뒤 각 사실이 소스로 지지되는지 계산한다.
14-2의 "문장 단위" 검증보다 더 세분화된 대안으로, 11주차 고도화 항목으로
남겨둔다. 문장 단위 검증이 이미 `[seg#N]` 인용과 결합돼 있어 기본
설계로는 문장 단위면 충분하다는 것이 본 설계의 판단이다.

## 14-4. Judge 프롬프트 설계 --- G-Eval 방식 차용

G-Eval(Liu et al., arXiv:2303.16634)은 단순 예/아니오 판정 대신
Chain-of-Thought로 근거를 단계별로 작성시킨 뒤 최종 판정을 내리게 하는
form-filling 방식을 써서 사람과의 상관도를 크게 높였다(요약 평가
Spearman 0.514, 기존 지표 대비 우수). 본 설계의 judge 프롬프트도 이
방식을 따른다 --- "이 문장이 [seg#N]과 일치하는가?"를 곧바로 묻지
않고, "① 문장이 주장하는 내용을 먼저 요약 → ② 인용된 seg#의 실제 내용을
요약 → ③ 두 내용이 일치하는지 근거를 들어 판정"의 3단계 프롬프트로
구성한다.

# 15. LLM-judge 신뢰성 리스크 --- 자기강화 편향 (Self-Enhancement Bias)

**핵심 경고: 리포트 생성과 채점(judge)에 같은 LLM을 쓰면 자기강화 편향이
생긴다는 것이 LLM-as-judge 연구에서 반복적으로 확인된 문제다.** 아래는
이 문제를 뒷받침하는 5편의 근거와 완화책이다.

## 15-1. 원조 정의 --- Zheng et al., MT-Bench (arXiv:2306.05685)

GPT-4를 judge로 써서 챗봇 응답을 비교 평가하는 방법론을 제안하면서,
position bias·verbosity bias와 함께 self-enhancement bias(judge가 자기
계열 모델의 출력을 더 선호하는 경향)를 실증적으로 처음 정의·검증한 원조
논문이다. Position swap·reference-guided grading 등의 완화책도 함께
제시했으며, GPT-4 판정과 인간 판정의 일치율은 80%대로 보고됐다.

## 15-2. 편향의 정량 측정 --- DBG Score (Chen et al., arXiv:2506.02592)

단순히 judge가 자기 응답에 준 점수와 남의 응답에 준 점수의 차이(승률
차이)를 편향의 척도로 쓰면, 응답 품질 자체의 차이와 편향이 뒤섞인다는
문제를 지적한다. 대신 여러 강한 LLM(논문에서는
GPT-4o-mini·Gemini-1.5-Flash·DeepSeek-V3)의 판정을 합의한 gold
judgment를 기준선으로 두고, judge의 판정과 gold judgment의 차이(DBG
score)만을 편향의 척도로 쓴다.

이 논문에서 본 프로젝트에 특히 중요한 실험 결과 두 가지:

-   **사전학습 모델과 post-training(instruct) 모델 모두 자기선호 편향을
    보이며, post-training이 편향을 더 키우지도 않는다.** 즉 "instruct
    모델을 쓰면 이 문제가 없어진다"는 가정은 틀렸다.
-   **모델 크기가 커질수록 편향은 작아지는 경향이 있다** (예:
    Qwen2.5-0.5B-Instruct의 DBG score 41.7% vs Qwen2.5-14B-Instruct
    2.1%). 다만 완전히 사라지지는 않는다.

*주의: 이 수치들은 AlpacaEval·TruthfulQA·WMT19 등 해당 논문의
벤치마크에서 보고된 값이며, 본 프로젝트의 AAR 리포트 채점 상황에 그대로
전이된다는 보장은 없다. "선행연구가 보고한 경향"으로만 인용한다.*

## 15-3. 자기개선 루프에서의 증폭 --- Pride and Prejudice (Xu et al., arXiv:2402.11436)

LLM이 self-refine(자기 출력을 스스로 반복 수정)을 거칠수록 자기 출력에
대한 과대평가가 누적된다는 것을 Bias·Dskew 지표로 정량화했다.
GPT-4·Gemini 모두 반복할수록 편향이 커졌고, 외부 피드백이나 더 큰 모델을
쓰는 것이 완화책이라고 결론짓는다. 본 설계에는 반복 개선 루프(예:
"리포트를 스스로 재검토해 수정"하는 단계)를 넣을 계획이 없지만, 넣게 될
경우 이 문제가 특히 위험해진다는 점을 사전에 경고해둔다.

## 15-4. 프롬프트 압력(tone)에 따른 환각 증폭 --- Ghost-100 (Jiang et al., arXiv:2604.18803)

negative-ground-truth 이미지(정답이 원천적으로 존재하지 않는 이미지)에
대해 VLM에 주는 프롬프트의 어조(tone)를 L1(중립)~L5(강압)로 올릴수록,
대부분의 모델에서 환각 커밋 비율(H-Rate)과 심각도(H-Score)가 증가한다는
것을 9개 오픈소스 VLM으로 실증했다(모델별 편차가 크고, 일부는 비단조적
--- 예: Gemma-4-E4B-IT는 중간 tone에서 정점을 찍고 최고 tone에서 오히려
급감).

**본 설계에 대한 직접적 시사점:** Ghost-100의 L4/L5 프롬프트는 "확신하지
못하면 PRESENT(있음)로 표시하라"는 식으로 불확실성을 긍정 판정 쪽으로
유도하는데, 이런 어투가 환각을 늘린다는 것이 이 논문의 핵심 발견이다.
따라서 14장의 Groundedness judge 프롬프트는 **정반대 방향**을 명시해야
한다 --- "인용된 seg#와 문장 내용이 정확히 일치한다고 확신할 수 없으면
grounded=false(불일치)로 보수적으로 판정하라"는 원칙을 프롬프트에
못박는다.

## 15-5. G-Eval 자체에서도 보고된 자기선호

14-4에서 채택한 G-Eval 방식 자체도, GPT-4가 GPT-3.5 출력을 인간
평가자보다 후하게 평가하는 self-preference 경향을 논문 내에서 스스로
보고하고 있다. CoT 기반 판정 방식을 쓴다고 해서 자기강화 편향이 자동으로
해소되지 않는다는 것을 재확인시켜주는 대목이다.

## 15-6. 완화책 및 한계 명시 전략 (우선순위순)

1.  **이상적:** 리포트 생성 LLM과 judge LLM을 다른 계열로 분리한다(예:
    생성은 Qwen2.5-Instruct, 채점은 다른 패밀리의 오픈소스 모델). GPU
    여유가 확인되면 최우선 채택.
2.  **리소스 제약 시 최소 조치:** (a) 결과보고서 9장 스타일로 "동일
    모델을 생성·채점에 썼다"는 한계를 명시하고, (b) 평가 질의셋 중
    일부(예: 50~100개 리포트 문장)를 사람이 직접 Groundedness 채점해
    LLM-judge와의 일치율을 함께 보고한다. 20개 수준의 표본은 일치율의
    95% 신뢰구간이 ±20%p에 달해 해석이 어려우므로 최소 50개를 권장하고,
    불가하면 신뢰구간을 병기한다.
3.  **Judge 프롬프트 설계 원칙:** 불확실할 때 보수적으로(=grounded
    아님으로) 판정하도록 명시한다(15-4의 교훈을 반대 방향으로 적용).
4.  **(여유 시) 교차검증:** 동일 리포트를 계열이 다른 오픈소스 judge
    모델로 한 번 더 채점해, 원래 judge와의 판정 차이를 자기편향의
    근사치로 보고한다(DBG score 방식의 축소 적용).

# 16. 예상 비판과 방어 논리 (9장 스타일 확장)

**"생성 LLM과 judge LLM이 같아서 평가가 무의미하다"는 비판에 대해:**
15-6의 완화책, 특히 사람 스팟체크와의 일치율을 정량적으로 함께
보고함으로써 "동일 모델 사용의 한계를 인지하고 있으며, 그 영향을
정량적으로 통제·보고했다"는 능동적 서사로 대응한다. 이는 기존
7-9절(SG-VLM의 한계 활용)·9-2절(신규성 포지셔닝)과 같은 톤이다 ---
한계를 숨기지 않고 측정 가능한 형태로 제시하는 것 자체가 방어 논리가
된다.

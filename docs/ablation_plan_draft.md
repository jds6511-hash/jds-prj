## 실험 1 — 세그먼트 길이 ablation (5초 → {3, 5, 8, 10}초)

### 1-1. 가설

- **짧은 세그먼트(3초)**: 검색 단위가 잘아져 정답 구간과 격자 정렬이 좋아지고(hit 시 시점 점프 정밀도↑), 한 세그먼트에 이질적 장면이 섞일 확률이 줄어 캡션·자막의 의미 순도가 올라간다. 반면 (a) 자막이 발화 단위보다 잘게 쪼개져 문맥이 깨지고(8-1 오버랩 귀속의 경계 케이스 증가), (b) 세그먼트 수가 5/3배로 늘어 top-k 안에 정답이 들 확률이 구조적으로 내려가며(같은 k에서 후보 풀 확대), (c) 캡션 생성 대상 프레임 수가 늘어 비용이 증가한다.
- **긴 세그먼트(8·10초)**: 자막 문맥이 온전히 담겨 자막형 질의에 유리하고 세그먼트 수 감소로 hit@k가 명목상 오르기 쉽다. 반면 한 세그먼트에 복수 장면이 섞여 대표 프레임 1장이 세그먼트를 대표하지 못하는 비율이 늘고(2장 SMART STC의 전제 약화), 정답 구간(수 초 단위)보다 검색 단위가 커져 IoU 보조지표가 나빠진다.
- 원 근거: 5초는 EventFormer의 DiDeMo window 비교에서 온 차용값(v2 1장)이며, 9-3이 명시하듯 "본 프로젝트 데이터(한국어 vlog·단일 영상)에서의 전이 보장이 없어" 자체 재확인이 이 실험의 존재 이유다.

### 1-2. 종속 변수

- 주지표: dev hit@1/5/10, MRR (세그먼트 인덱스 기반, 8-3)
- 보조: IoU@0.5/0.3 — **길이 간 비교에서 특히 중요**. hit@k는 세그먼트 수가 다르면 난이도가 달라지는 지표이므로(10초는 세그먼트 수가 절반 → 같은 hit@5라도 더 쉬운 문제), 격자와 무관한 IoU 보조지표를 반드시 병기해 해석한다. 필요 시 "hit@k를 k×(5/seg_len)로 보정한 참고치"를 각주로 제시.

### 1-3. 통제 변수와 핵심 설계 결정: α 재탐색 여부

임베딩 모델(KURE-v1), static_threshold(0.05), 캡션 프롬프트, VLM 설정(4bit·max_pixels·rep_penalty)은 전부 고정. 유일한 쟁점은 **α를 길이별로 dev 재탐색할지, 5초에서 확정한 α로 고정할지**다.

- **고정 논거**: 변인이 세그먼트 길이 하나로 깨끗해지고, 실행 비용이 준다(길이당 grid search 생략). "5초 확정 시스템에서 길이만 바꾸면?"이라는 민감도 분석으로 해석이 단순.
- **재탐색 논거**: 길이가 바뀌면 자막 커버리지(긴 세그먼트일수록 무발화 세그먼트 비율↓)와 캡션의 대표성이 함께 바뀌어 최적 α 자체가 이동한다. α를 5초 최적값에 고정하면 다른 길이가 "자기 최적이 아닌 α"로 평가받아 **5초에 유리한 편향 비교**가 된다 — 심사에서 바로 지적당할 지점.
- **권장안: 길이별 dev grid search 재탐색.** 근거: (a) 이 ablation의 질문은 "각 길이의 최선끼리 비교했을 때 5초가 여전히 좋은가"(9-3의 취지)이지 α 민감도가 아니다. (b) grid search는 M6에서 캐시된 임베딩으로 수 초~수 분이면 끝나 추가 GPU 비용이 사실상 0이다. (c) dev에서만 하므로 누수 없음(9-1). 결과 표에 길이별 α*를 함께 보고하면 "길이에 따른 모달리티 기여 변화"라는 부수 분석도 얻는다. 단 결과 표에 "α=α_5s 고정" 열을 참고로 병기하면 양쪽 해석을 다 커버할 수 있다(추가 비용 0).

### 1-4. 재실행 범위·절차·비용

seg_len은 M1의 분할 자체를 바꾸므로 **M1→M2→M3→M4 전체 재실행**이 필요하다 (프레임·자막 귀속·캡션·임베딩 전부 길이에 종속).

절차 (길이 L ∈ {3, 8, 10}마다):
1. config 사본 생성: `seg_len_sec: L`만 변경 (예: `config_seg3.yaml`)
2. 산출물 분리: `work/{video_id}_seg{L}/` 형태로 출력 경로 분리 (5초 산출물 덮어쓰기 금지 — 멱등 원칙과 별개로 비교용 보존 필요) → **[검토 필요]** 현재 paths 구조상 video_id에 suffix를 붙이는 방식이 최소 변경인지 확인
3. `m1 → m2 → m3 → m4` 순차 실행 → verify: n_segments == ceil(duration/L), 캡션 빈 문자열 0건
4. gt_seg_idx 재산출: **질의 라벨의 gt_start/gt_end는 초 단위 타임스탬프라 그대로 재사용 가능.** `m6_evaluate.py`의 `derive_gt_seg_idx(gt_start, gt_end, n_segments, seg_len)`이 이미 seg_len 인자화되어 있으므로 queries.jsonl의 gt_seg_idx 필드를 무시하고 L 기준으로 재계산하면 된다. **재라벨링 불필요** — 이것이 초 단위 라벨을 원본으로 둔 설계의 이득.
5. dev grid search → 길이별 α* 및 hit/MRR/IoU 기록

비용 (RTX 3060 6GB, 26분 영상 1개 실측 기준: M2 ~50분, 캡션 314프레임 ~20-30분):
- 세그먼트 수는 3초 ≈ 520개 / 8초 ≈ 195개 / 10초 ≈ 156개 (26분 기준). M2는 프레임 순회가 지배적이라 길이와 무관하게 영상당 ~50분 수준, 캡션은 세그먼트 수에 비례(3초 ≈ 35-50분, 8·10초 ≈ 12-20분). Whisper 전사는 세그먼트와 무관(재귀속만 재실행).
- **영상 1개 × 길이 3종 추가 ≈ 4.5~6 GPU시간.** 영상이 N개면 선형 증가 — dev 영상 전체가 아니라 **dev 영상 1~2개로 축소 실행**하는 것이 현실적 (아래 우선순위 절 참조).

### 1-5. 해석 시나리오

- 5초가 최고 또는 동률 → "차용 가정이 본 데이터에서 재확인됨"으로 결론, 본 실험 자체가 9-3 방어 논리의 완성.
- 3초 우세 + 장면형 질의에서 특히 → 짧은 단위의 캡션 순도 효과. 단 hit@k 난이도 차이(1-2절) 보정 후에도 유지되는지 확인하고, IoU 보조지표 동반 상승 여부로 교차 검증.
- 8·10초 우세 → 자막 문맥 보존 효과 가설 지지. 유형별 분해에서 자막형 주도인지 확인. 자막형 주도라면 "검색 단위는 발화 단위에 가까울수록 유리"라는 향후과제(가변 분할, EventFormer)로 연결.
- 길이별 α*가 체계적으로 이동(예: 길수록 α*↑) → 모달리티 기여가 세그먼트 길이의 함수라는 부수 발견으로 보고.

## 실험 2 — static_threshold ablation (0.05 → 분포 기반 후보)

### 2-1. 가설

정적 치환(8-4: s_cap_norm ← s_sub_norm)은 "정적 구간 캡션 = 환각 노이즈"일 때 검색을 돕고, "정적이지만 캡션이 정확·유익"할 때(예: 고정 카메라로 물건을 보여주며 설명하는 vlog 장면) 장면형 질의의 시각 단서를 버려 해가 된다. 즉:
- threshold ↑ (치환 확대) → 자막형 질의 안정화, 장면형 질의 성능은 캡션 품질에 따라 양방향.
- threshold = 0 (치환 off) → proposed의 장면 기여 최대화, 환각 노이즈 유입 최대화. **8-4 말미가 명시한 "예외 처리 on/off ablation"이 바로 이 극단점**이므로 후보에 반드시 포함.

### 2-2. 실측 분포 활용 — M2 재실행 불필요 (핵심 이점)

DESIGN_SPEC 3-1의 설계 의도 그대로: **motion_score가 segments.json에 이미 기록돼 있으므로 is_static은 임의 threshold로 재판정 가능하다.** M2(영상당 ~50분)를 다시 돌릴 필요가 전혀 없다 — `is_static = motion_score < thr`를 M5 로드 시점(static_mask 구성)에 재계산하면 된다. M3·M4도 재실행 불필요(캡션·임베딩은 threshold와 무관, 4-2 확정 로직대로 정적 세그먼트도 캡션은 이미 생성돼 있음). **이 실험은 사실상 CPU-only, 스윕 전체가 수 분.**

구현 메모: 현재 M5가 segments.json의 저장된 `is_static`을 읽는다면, threshold 인자로 motion_score에서 mask를 재계산하는 경로 하나만 추가(코드 변경 최소; 저장 필드 갱신 없이 평가 시점 재판정이 멱등성에도 안전). **[검토 필요]** rep_frame fallback(정적 시 중간 프레임)은 M2 산출이라 재판정으로 바뀌지 않음 — 캡션이 "0.05 기준 대표 프레임"으로 생성된 채 threshold만 바뀌는 비대칭이 있음을 한계로 명시할 것.

### 2-3. 후보 threshold 설계 — 분포 기반 (2026-07-10 재설계, 실촉 데이터로 갱신)

> 이전 초안은 spiderman_trailer(영화 예고편, 장르 부적합으로 전체 제외됨 — 2026-07-10) 1개 영상의
> 실측치로 "트레일러는 고모션이라 threshold 캘리브레이션에 부적합"이라는 가설만 세운 상태였다.
> 이제 현재 6개 영상 인덱싱이 모두 끝나, 실제 dev 3개·test 2개 영상의 motion_score 분포를 직접

**원래 가설의 방향은 틀렸다**: 트레일러(고모션)가 이례적으로 낮은 static 비율을 만드는 게 아니라,
**다큐멘터리(kheritage, 유물·발굴 아카이브 영상)가 정반대로 이례적인 저모션 이상치**다 —
median이 다른 4개 영상(0.13~0.23)의 절반 수준(0.074)이고, thr 0.05에서 static 비율이 32%로
나머지 4개(0~4.6%)와 자릿수가 다르다. 나머지 4개 영상(Wilderness/gwaktube/panibottle/gemini)은
장르(생존 vlog·여행 vlog·홍보 talking-head)와 무관하게 median이 0.13~0.23 범위에 모여 있어,
**현행 절대값 0.05는 이 4개 영상에서는 사실상 "거의 off"이고, kheritage류 다큐(정적 카메라+아카이브
클로즈업이 많은 장르)에서만 실질적으로 발동한다.**

- 후보: **{0 (off), 0.05 (현행), P10, P25, P50}** — 여전히 절대값이 아니라 **영상별 motion_score
  분위수**로 잡는다. 이번엔 5개 영상 실측이 있으므로 영상별 절대값 예시를 함께 병기한다:
  - Wilderness: P10 0.100 / P25 0.138 / P50 0.174
  - kheritage: P10 0.031 / P25 0.045 / P50 0.074
  - gwaktube: P10 0.071 / P25 0.098 / P50 0.149
  - panibottle: P10 0.066 / P25 0.101 / P50 0.134
  - gemini_promo: P10 0.161 / P25 0.203 / P50 0.229
- **주의(갱신)**: dev 3개 영상만으로 분위수를 잡으면 kheritage 저모션 이상치가 dev 평균을 아래로
  끌어당겨, 절대 threshold를 dev 평균으로 캘리브레이션할 경우 gwaktube/Wilderness류 일반 vlog에서는
  과소 발동(치환이 너무 안 걸림), kheritage류 다큐에서는 여전히 과잉 발동 위험이 남는다 — **영상 장르
  혼합(vlog + 다큐)이 dev 구성에 실제로 존재하므로, 분위수 기반 threshold가 절대값보다 일반화에
  유리하다는 이전 결론이 실측으로 재확인됨.** 절대값 1개를 config에 고정해야 하는 최종 시스템 제약
  (6장: "dev에서 1회 보정 후 고정")과의 긴장은 여전히 남아 있음 — **[검토 필요]** 분위수 방식 채택 시
  config 스키마 변경 필요.

### 2-4. 평가 방법

- threshold 스윕 × **α 고정**(현행 dev 확정 α). 치환은 α와 상호작용하지만(치환된 세그먼트는 score = s_sub_norm이 되어 α 무관), 스윕 축을 threshold 하나로 유지해야 해석이 깨끗하다. 민감도 확인용으로 α ∈ {α*, 0.5} 2점 정도만 병행 가능(비용 무시 가능 수준).
- **장면형 질의 부분집합 집중 분석**: 치환은 캡션 점수를 덮어쓰는 조작이므로 효과는 장면형에서 가장 크게 나타나야 한다. gt 세그먼트가 정적으로 재판정된 질의만 골라 per-query rank 변화(3-4 per_query 형식 재활용)를 추적하면 "치환이 도운/해친 질의" 사례 표가 바로 나온다.
- 자막형 질의는 sanity check(치환의 부작용이 없어야 정상).

| 0.08 | 5.7% | 0.833 | 0.619 | 0.750 |
| 0.10 | 9.9% | 0.750 | 0.615 | 0.500 |
| 0.14 | 27.4% | 0.583 | 0.531 | **0.000** |

**0.08~0.10 사이에서 급격한 성능 붕괴**(장면형 hit@5가 0.75→0.50→0.00)를 확인 — 2-5절 해석 시나리오 중 "장면형에서 치환 손해" 시나리오가 소규모로 이미 관측됨. 현행 0.05는 안전 구간이나, 표본(dev 12개, 장면형 4개)이 너무 작아 최적점(0.02~0.05 구간 어디)까지는 특정 불가. **사용자 결정: 0.05 유지, 60개 질의 확보 후 정식 재측정.** 이 사전 실측은 실험 2의 방향성 검증이지 최종 결론이 아니다 — 정식 실행 시 P10/P25/P50 후보와 함께 재확인.

### 2-5. 해석 시나리오

- off(0)가 최선 → 캡션 환각이 실데이터에서 문제가 아니었다는 뜻. 치환 로직은 "안전장치였으나 발동 불필요"로 보고하고 threshold는 보수적으로 유지.
- 중간 분위수(P10~P25)가 최선 → 8-4 설계 의도 그대로 검증 성공. "환각 노이즈의 공학적 제어"(9-2 신규성 서사)의 정량 근거 확보 — **보고서 기여 주장과 직결되는 결과**.
- P50 이상에서 단조 상승 → 캡션 품질 자체가 낮다는 신호. 실험 3(프롬프트)과 연결해 "치환 확대는 캡션 개선의 대증요법"으로 해석.
- 장면형에서 치환 손해 + 자막형 무변화 → 치환이 proposed의 장면 기여를 깎는다는 8-4 말미의 우려 실증 → threshold 하향 권고.

## 실험 3 — 캡션 프롬프트 다양화

### 3-1. 후보 프롬프트 (현행 포함 4종 + 선택 1종)

현행(P0): "이 장면을 한 문장의 한국어로 객관적으로 묘사하라. 화면에 보이지 않는 것은 쓰지 마라."

| ID | 프롬프트(안) | 변형 축 |
|----|----|----|
| P1 상세형 | "이 장면에 보이는 인물의 행동, 주요 객체, 배경을 한국어 두 문장 이내로 객관적으로 묘사하라. 화면에 보이지 않는 것은 쓰지 마라." | 디테일 수준↑, 2문장 허용 |
| P2 객체 중심 | "이 장면에 보이는 주요 객체와 장소를 한국어 한 문장으로 나열하듯 묘사하라. 추측하지 마라." | 객체/장소 명사 위주 — 명사구 질의와의 매칭 가설 |
| P3 행동 중심 | "이 장면에서 인물이 무엇을 하고 있는지 한국어 한 문장으로 묘사하라. 인물이 없으면 화면의 상태 변화를 묘사하라. 보이지 않는 것은 쓰지 마라." | 동사/행동 위주 — 장면형 질의("~하는 장면")와의 매칭 가설 |
| (선택) P4 간결형 | "이 장면을 한국어 10단어 이내의 짧은 구로 요약하라." | LMR(7-2(3))의 간결형 극단 — 환각 여지 최소화 대조군 |

근거: LMR(v2 7-2(3))이 상세형~간결형 다중 지시문 캡션으로 성능을 얻었고, 구현가이드가 이를 "11주차 고도화 실험으로 응용"하라고 명시. 질의셋의 장면형 질의가 "~하는 장면"(행동) 꼴인지 "~가 있는 장면"(객체) 꼴인지 분포를 먼저 확인하고 P2/P3 우선순위를 정한다.

### 3-2. 가설

- P1(상세): 정보량↑로 복합형·장면형 recall 개선, 단 환각 표면적도↑ (문장이 길수록 근거 없는 서술 혼입 위험) — 정적 치환(실험 2)과 상호작용.
- P2(객체) vs P3(행동): 질의 유형 분포와의 정합이 성능을 가른다. 장면형 질의가 행동 서술 위주라면 P3 우세 예상.
- 극단 가설: 프롬프트 간 차이가 hit@k에서 유의하지 않다면 "캡션 스타일보다 α 결합·치환 같은 후단 설계가 지배적"이라는 결론도 그 자체로 보고 가치가 있음.

### 3-3. 재실행 범위·비용

프롬프트는 M3(b) 캡션 생성부터 바뀌므로 **M3(b) + M4 재실행** (M1·M2·M3(a) Whisper는 재사용 — segments.json의 subtitle 필드 보존 필요, `--force` 시 필드 단위 덮어쓰기 동작 확인 **[검토 필요]**).

- 비용: 캡션 314프레임 ≈ 20-30분 실측 → 26분 영상 1개 × 프롬프트 1종 ≈ 25분 + M4 임베딩(분 단위). **프롬프트 3종 추가 ≈ 영상당 1.5시간.** 세 실험 중 실험 2보다 비싸고 실험 1보다 싸다.
- 산출물 분리: `work/{video_id}_p{N}/` 또는 segments.json에 caption을 프롬프트별 파일로 분리 저장 — 실험 1과 동일한 경로 규칙 사용 권장.
- α: 프롬프트별 dev 재탐색(grid search 비용 0, 실험 1의 권장안과 일관).

### 3-4. 캡션 품질 중간 지표 (검색 성능과 분리해 자동 측정)

검색 지표만 보면 "왜 좋아졌는지"를 알 수 없으므로, 프롬프트별로 다음을 자동 집계해 성능과 나란히 표로 제시한다 (전부 CPU 스크립트, 분 단위):

1. **평균 길이** (문자 수·어절 수) — 디테일 수준의 조작 확인(manipulation check)
2. **반복률** — (a) 캡션 내 n-gram 반복 비율(rp=1.3으로 억제한 반복 붕괴의 재발 감지), (b) 영상 내 캡션 간 중복도(유니크 캡션 비율, 인접 캡션 간 자카드 유사도) — 캡션이 서로 비슷하면 변별력이 없어 검색에 무용
3. **명사/동사 밀도** — 형태소 분석(Kiwi 등)으로 어절 대비 명사·동사 비율 — P2/P3 조작 확인 겸 "질의 어휘와의 품사 정합" 분석 재료
4. **캡션-질의 어휘 겹침** — dev 질의 텍스트와의 명사 중복률(참고치)
5. (참고) 캡션 임베딩 분산 — emb_cap의 평균 pairwise 코사인. 낮을수록 변별력 있는 인덱스

해석 연결: 예컨대 P1이 성능↑ + 반복률↓ + 명사 밀도↑라면 "정보량 증가가 원인"으로, P1이 성능↑인데 중간 지표 차이가 없다면 "임베딩 공간에서의 우연적 정합"을 의심하고 질의별 사례 분석으로 내려간다.

### 3-5. 해석 시나리오

- 특정 프롬프트가 장면형에서만 개선 → "캡션 스타일-질의 유형 정합" 발견. 보고서 10주차 유형별 분석 표에 프롬프트 축 추가.
- P1(상세)이 개선 + 환각(실험 2의 치환 발동 시 이득)도 증가 → "디테일-환각 트레이드오프"로 보고, 최종 config는 hit@5 기준 선택.
- 전 프롬프트 동률 → 캡션 프롬프트는 robust하다는 결론. 현행 P0 유지 + LMR식 다중 프롬프트 앙상블은 향후과제로 격하.

## 실행 우선순위 권장안

제약: 라벨된 질의셋 규모(dev 질의 수십 개 수준 — 유형별 분해 시 셀당 표본이 작아 큰 차이만 신뢰 가능), 로컬 RTX 3060 6GB(인덱싱 영상당 M2 ~50분 + 캡션 ~25분).

| 순위 | 실험 | 근거 |
|---|---|---|
| **1** | **실험 2 (static_threshold)** | GPU 비용 ~0 (motion_score 재판정만, 3-1 설계 의도의 회수), 스윕 전체가 몇 분. 결과가 9-2 신규성 서사("환각의 공학적 제어")와 직결. 2-3절 실측(2026-07-10, 5개 영상)상 현행 0.05는 4개 영상에서 사실상 미발동(0~4.6%)이나 kheritage(다큐 장르)에서만 31.8% 발동 — **장르에 따라 유효성이 갈리는 상태가 실측으로 확인돼 가장 시급.** dev 분포 히스토그램 확인은 완료, 다음 작업은 분위수 기반 threshold 스윕 실행. |
| **2** | **실험 3 (캡션 프롬프트)** | 영상당 ~1.5 GPU시간(M3(b)+M4만). 중간 지표가 자동 측정이라 분석 비용도 낮음. 실험 2 결과(캡션 품질 신호)를 받아 후보를 조정할 수 있어 순서상 뒤가 유리. |
| **3** | **실험 1 (세그먼트 길이)** | M1부터 전면 재실행으로 최고 비용(영상당 길이 3종 ≈ 4.5~6 GPU시간). 단 9-3 방어 논리상 **생략 불가** — 비용 절충: dev 영상 1~2개 + 길이를 {3, 5, 10} 3종으로 축소(8초는 5·10 결과가 비단조일 때만 추가). 밤샘 배치로 실행. |

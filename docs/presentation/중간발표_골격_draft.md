## 1부. 발표 스토리라인 (본편 14장 + 백업 2장)

### 슬라이드 1 — 표지
- **핵심 메시지:** 멀티모달 LLM 기반 단일 영상 장면 검색 — baseline(자막) vs proposed(자막+장면 캡션)의 정량 비교.
- 담을 내용:
  - 제목: 영상 장면 검색 및 AAR 보고서 자동 생성 시스템 (중간발표)
  - 9주차
- 그림: 없음 (또는 데모 스크린샷 1장 티저)

### 슬라이드 2 — 문제 정의: 왜 영상 장면 검색인가
- **핵심 메시지:** 긴 영상에서 "원하는 장면"을 자연어로 바로 찾는 것은 여전히 수작업 스크러빙에 의존하며, 사후검토(AAR) 활용의 병목이다.
- 담을 내용:
  - 입력 = 업로드한 단일 mp4 + 자연어 질의 → 출력 = 점프 가능한 top-k 세그먼트 (정수 초 `[[시작,끝],...]` + 자막)
  - 자막(STT)만으로 검색하면 "말하지 않고 보여주기만 한 장면"(장면형 질의)을 놓친다 — 이것이 본 연구의 출발점
  - 확장 목표: 질의 없이 영상 전체를 훑어 근거 인용(`[seg#N]`)이 달린 AAR 리포트 자동 생성 (제2부, 계획 단계)
- 그림: 문제 상황 개념도 (긴 타임라인 + "이 장면 어디였지?" 질의 예시)

### 슬라이드 3 — 문제 설정(scope)
- **핵심 메시지:** corpus-level VCMR이 아니라 "단일 업로드 영상 내 segment retrieval"로 문제를 고정하고, 학습 없는(비학습) 임베딩 검색으로 14주 안에 검증한다.
- 담을 내용:
  - 연구 질문: 자막 단독(baseline) 대비 자막+장면 캡션 결합(proposed)이 검색 성능을 얼마나·어떤 질의 유형에서 개선하는가
  - 제약 조건: 한국어 vlog, 온프레미스(클라우드 API 배제), 학습 없음(frozen 모델만)
  - 안 하는 것: 검색기 학습(PREM), 오디오 제3 모달리티(SMART), 가변 분할(EventFormer 학습부) — 향후과제로 명시
- 표: 할 것 / 안 할 것 2열 표 (가이드 "한눈에 보기" 축약)

### 슬라이드 4 — 관련 접근과 한계
- **핵심 메시지:** 선행연구는 대부분 학습 기반이라 그대로 재현하기 어렵고, 본 연구는 거기서 "학습 없이 차용 가능한 설계 결정"만 골라 통합한다.
- 담을 내용:
  - TVR·PREM·EventFormer·SMART: contrastive/adversarial/LoRA 학습 필요 → 성능 상한선·차별점 근거로만 인용
  - Video Enriched RAG: 자막+캡션 정렬 텍스트 블록이 최고 성능(Table 3: ASR HIT@1 0.741 vs 캡션 단독 0.65) — 본 설계의 청사진
  - LMR(arXiv:2405.12540)·SG-VLM: "frozen VLM 캡션을 검색에 활용"하는 가장 가까운 학술 선례. SG-VLM은 무거운 구조화 캡션의 이득이 제한적임을 보고 → 가벼운 단일 프레임 캡션 선택의 근거
- 표: 단계별 참조 논문 매핑 표 (가이드 0-1 축약: 분할=EventFormer, 프레임=SMART, 임베딩=Video Enriched RAG, 결합=PREM·Chrono, 평가=TVR)

### 슬라이드 5 — 제안 방법: 파이프라인 개요
- **핵심 메시지:** ①5초 분할 → ②대표 프레임 → ③자막+캡션 생성·임베딩 → ④정규화+α 가중합 검색 → ⑤평가 → ⑥데모, 모듈 간 통신은 파일로만(재현성).
- 담을 내용:
  - M1~M7 모듈 다이어그램 (설계서 5장 실행 순서 그림화)
  - 각 모듈 = 독립 CLI + 멱등(idempotent), 중간 산출물 전부 파일(JSON/JSONL/NPY)로 보존 → 디버깅·부분 재실행 용이
  - 모델 스택: Whisper large-v3(STT) / Qwen2.5-VL-3B 4bit(캡션) / KURE-v1(임베딩) — 전부 오픈소스·온프레미스
- 그림: **파이프라인 전체 다이어그램 (필수, 이 발표의 중심 그림)**

### 슬라이드 6 — 설계 결정 ①: 5초 고정 세그먼트 + 대표 프레임
- **핵심 메시지:** 5초 고정 분할은 단순하지만 근거 있는 선택이고, 대표 프레임은 "가장 동적인" 프레임을 골라 캡션 정보량을 높인다.
- 담을 내용:
  - 왜 5초 고정: EventFormer가 세 분할 전략 비교에서 DiDeMo(5초 단위)의 고정 window가 최우수 → 근거 있는 기본값. 11주차에 3/5/10초 자체 ablation으로 재확인 예정
  - 왜 대표 프레임 선택: SMART STC 차용 — 프레임 차분 L2 norm + 가우시안 평활 후 argmax (중간 프레임보다 정보량 많은 프레임)
  - 정적 세그먼트(차분 < threshold 0.05)는 중간 프레임 fallback + `is_static` 플래그 기록 (슬라이드 8에서 재사용)
  - 경계 걸친 발화: Whisper word-level timestamp로 "더 많이 걸친 세그먼트 귀속 + 경계 문장 양쪽 중복 허용" (가이드 8-1)
- 그림: 5초 격자 + 대표 프레임 선택 예시 (실제 영상 프레임 차분 그래프면 더 좋음)

### 슬라이드 7 — 설계 결정 ②: 자막+캡션 이중 인덱스
- **핵심 메시지:** VLM은 캡션 "생성기"로만 쓰고(frozen), 검색은 자막·캡션·질의를 동일 임베딩 모델의 코사인 유사도로 수행한다.
- 담을 내용:
  - 왜 이중 인덱스: 자막은 "말한 것", 캡션은 "보이는 것" — 상호 보완 (Video Enriched RAG Table 3이 결합 우수성 보고)
  - 왜 frozen VLM + 임베딩 분리: LMR·SG-VLM 선례. VLM으로 직접 검색하지 않아 계산 비용·재현성 문제 회피
  - 왜 동일 임베딩 모델(KURE-v1): 자막·캡션·질의가 통일된 의미공간에 있어야 코사인 비교가 성립 (가이드 7-8). KURE vs BGE-M3는 dev셋에서 장면형 질의 분리 검증 후 확정 — 예비 비교(소규모 확정 라벨 n=6~12, 참고용): 두 모델 모두 baseline 대비 proposed 우위 방향은 일치, KURE가 test에서 근소 우위 — `[정식 60개 질의 확보 후 확정 수치로 갱신]`
  - 캡션 프롬프트 1종 고정: "이 장면을 한 문장의 한국어로 객관적으로 묘사하라. 화면에 보이지 않는 것은 쓰지 마라." — 캡션 언어 = 질의 언어 = 한국어 원칙
- 그림: 세그먼트 1개의 (대표 프레임 → 캡션, 오디오 → 자막) → emb_sub/emb_cap 2벡터 예시 (실제 데이터 사용 권장)

### 슬라이드 8 — 설계 결정 ③: 확정 연산 순서 — 정규화 → 정적 치환 → α 가중합
- **핵심 메시지:** 자막·캡션 유사도는 분포가 달라 그대로 합치면 α 실험 자체가 왜곡된다 — 연산 순서를 고정하는 것이 실험 타당성의 전제다.
- 담을 내용:
  - 순서(고정): ① 코사인 s_sub·s_cap 계산 → ② 각각 per-query min-max 정규화(단일 영상 범위) → ③ 정적 세그먼트는 s_cap_norm ← s_sub_norm 치환 → ④ `score = α·s_sub + (1-α)·s_cap`
  - 왜 정규화: 자막(구어체)과 캡션(VLM 생성문)의 코사인 분포가 달라, 예컨대 자막 0.6~0.8 vs 캡션 0.4~0.6이면 α=0.5여도 자막이 랭킹을 지배 (가이드 8-2)
  - 왜 치환(버리기 아님): 정적 구간은 VLM 캡션 환각 위험 → 캡션을 버리면 점수 상한 불연속이 생기므로, s_sub를 한 번 더 보게 치환해 스케일(상한 1.0)을 동적 세그먼트와 일치시킴 (가이드 8-4)
  - 왜 정규화 "후" 치환: 치환을 먼저 하면 정적 세그먼트의 원본 s_cap이 min/max 계산을 오염시킴
- 그림: 4단계 연산 순서 다이어그램 + 분포 불일치 히스토그램(모식도 가능)

### 슬라이드 9 — 시스템 데모: 웹 UI (라이브 데모 또는 영상)
- **핵심 메시지:** mp4 업로드 → 자동 인덱싱 → 채팅형 자연어 검색 → 클릭 한 번에 해당 초로 점프까지 end-to-end로 동작한다.
- 담을 내용:
  - 흐름: 업로드 → M1~M4 파이프라인 자동 실행(멱등: 재업로드 시 스킵) → 채팅창 질의 → top-3 결과(정수 초 창) → 클릭 시 플레이어 점프 + 자막 표시
  - 백엔드는 M5 `search()`를 그대로 import (재구현 금지 원칙 → 데모와 평가 코드가 동일 로직)
  - E2E 스모크 통과: 업로드→완료, 검색 top-3가 M5 CLI 결과와 동일함을 확인
- 데모: **라이브 데모 (필수).** 실패 대비 백업 스크린샷 3장(업로드/검색/점프) + 데모 영상 녹화 준비
- ⚠ 발표 전 확인: 데모용 영상 1개 사전 인덱싱, 질의 2~3개 리허설(자막형 1 + 장면형 1)

### 슬라이드 10 — 평가 설계: 질의셋과 지표
- **핵심 메시지:** 주지표는 세그먼트 인덱스 기반 Hit@k/MRR — 5초 고정 격자에서는 IoU@0.5가 구조적으로 불리한 케이스가 있어 IoU는 보조지표로만 병기한다.
- 담을 내용:
  - 질의셋: 목표 60개(자막형/장면형/복합형 각 20) — `[질의셋 규모·유형 분포 라벨링 완료 후 확정]`. 2인 이상 라벨링, 라벨러 간 일치율 병기 예정
  - 왜 Hit@k 주지표: 정답 구간이 격자 경계를 가로지르면(예: 3~7초) 어떤 5초 세그먼트도 IoU 0.5를 못 넘는 억울한 실패 발생 (가이드 8-3) → 정답을 세그먼트 인덱스로 라벨링해 Hit/Recall@{1,5,10}·MRR로 평가
  - IoU@0.5/0.3은 선행연구(TVR 계열) 비교용 보조지표
  - 질의 유형별 분리 집계 → "장면 결합이 어떤 질의에서 이득/손해인가"가 핵심 분석
- 표: 지표 정의 요약표 (주지표/보조지표 구분)

### 슬라이드 11 — 평가 설계: baseline 공정성과 dev/test 분리
- **핵심 메시지:** baseline은 동일 파이프라인의 α=1.0 특수 경우이며(별도 코드 경로 금지), α는 dev에서만 탐색해 test에 고정 적용한다 — 누수를 코드 수준에서 차단했다.
- 담을 내용:
  - baseline = `search(query, α=1.0)`: 정규화·정적 치환 인프라가 proposed와 완전히 동일 → 비교가 대칭적
  - α grid search: 0.0~1.0, 0.1 간격, dev셋 hit@5 기준, 동률 시 α 큰 값(자막 우선) — 전부 config에 고정
  - dev/test는 video_id 단위로만 분리(같은 영상의 질의가 갈라지면 누수) + 유형 비율 층화. M6는 test로 α를 재탐색하는 코드 경로 자체가 없음
  - 재현성: config.yaml이 단일 진실 공급원 — seed 42, 전 고정값 명시
- 표: config.yaml 핵심 설정 발췌 표 ("실험 설정" 절 = config 그대로)

### 슬라이드 12 — 확장: AAR 리포트 생성과 이중 평가 (M8·M9)
- **핵심 메시지:** 검색 인덱스(자막+캡션)를 재활용해, 모든 문장에 `[seg#N]` 인용이 강제된 AAR 리포트를 생성하고 Coverage·Groundedness 쌍으로 평가한다 — 코드는 구현·리뷰 완료, 실행은 judge 모델 확정 대기.
- 담을 내용:
  - 생성: Qwen2.5-7B-Instruct, map-reduce(60세그먼트 초과 시), `[seg#N]` 인용 강제 (V2Xum-LLM 선례)
  - 평가: Event Coverage Rate + Groundedness(환각률) — 상충 관계(precision/recall)라 반드시 쌍으로 보고. judge는 G-Eval식 3단계 CoT
  - judge 모델은 report 모델과 다른 패밀리 원칙(자기강화 편향 완화) — GPU 사양 확인 후 확정 예정
  - **명확히 구분:** 검색 코어(M1~M7)와 AAR(M8·M9) 모두 구현·TDD·3중 리뷰 완료(전체 77 테스트 통과, 스텁 LLM으로 검증) — 실제 GPU로 리포트를 생성·평가해 본 적은 아직 없음
- 그림: AAR 파이프라인 간단 다이어그램 (검색 코어 그림에 ⑦단계 추가 형태)

### 슬라이드 13 — 현재 진행 상황
- **핵심 메시지:** M1~M9 + 웹 UI까지 구현·테스트 완료(전체 77 테스트 통과), 정량 평가는 질의 라벨링 규모 확대 직후 실행 가능한 상태다.
- 담을 내용:
  - 완료: M1~M5(전처리·프레임·자막/캡션·인덱싱·검색), M6 평가 코드, M7 데모 + 웹 UI(E2E 스모크 통과), M8·M9 AAR(스텁 LLM 검증 완료, 실행은 judge 모델 확정 대기)

## 2부. 예상 질문 Top 5 + 답변 초안

> 발표자가 그대로 말할 수 있는 수준으로 작성. 괄호는 말하지 않는 근거 출처.

### Q1. 세그먼트를 5초 "고정"으로 자른 근거는? 장면 경계와 어긋나면 어떡하나?
**답변 초안:** "EventFormer가 contrastive convolution, K-means, 고정 window 세 가지 분할 전략을 비교했는데, 5초 단위 데이터셋인 DiDeMo에서는 고정 window가 가장 우수했습니다. 그래서 5초 고정을 '근거 있는 기본값'으로 채택했고, 절대적 선택이라고 주장하지는 않습니다. 11주차에 저희 데이터에서 3초/5초/10초를 dev셋으로 비교하는 ablation을 계획해 두었고, 경계와 어긋나는 발화는 word-level timestamp 기반 오버랩 귀속 규칙으로, 경계에 걸친 정답 구간은 IoU 대신 세그먼트 인덱스 Hit@k 주지표로 각각 흡수했습니다. 또 샷 경계와 어긋나 실패한 질의가 있는지 오류 분석에서 별도로 확인할 예정입니다." (가이드 1장, 8-1, 8-3, 9-3)

### Q2. 캡션을 왜 한국어로 생성하나? VLM은 영어 캡션 품질이 더 좋지 않나?
**답변 초안:** "저희 검색은 자막·캡션·질의를 반드시 같은 임베딩 모델로 인코딩해서 코사인 비교하는 구조입니다. 질의가 한국어이므로 캡션도 한국어로 맞춰야 임베딩 공간의 언어 불일치로 인한 매칭 저하를 피할 수 있습니다 — '캡션 언어 = 질의 언어' 원칙입니다. 영어 캡션 + 번역이나 cross-lingual 임베딩은 오류 단계를 하나 더 추가합니다. 대신 다국어에 강한 Qwen2.5-VL을 선택했고, 실제 GPU 스모크에서 한국어 캡션이 유효하게 생성됨(생성 실패 0건)을 확인했습니다. 임베딩도 한국어 특화 KURE를 1순위로 두고 dev셋에서 BGE-M3와 비교해 확정합니다." (가이드 7-8, 8-5; progress 스모크 결과)

### Q3. 질의 60개는 너무 작지 않나? 통계적으로 의미 있는 비교가 가능한가?
**답변 초안:** "규모의 한계는 인정합니다. 다만 세 가지로 보완했습니다. 첫째, 60개를 자막형/장면형/복합형 20개씩 층화해서 '어떤 유형에서 결합이 이득인가'라는 핵심 질문에는 유형별 20개로도 방향성 있는 답을 얻을 수 있게 설계했습니다. 둘째, dev/test를 영상 단위로 분리하고 α 탐색을 dev로만 제한해, 작은 데이터에서 특히 위험한 과적합·누수를 차단했습니다. 셋째, 집계 수치만이 아니라 per-query 랭크를 전부 남겨 질의별 사례 분석(어떤 질의에서 장면 결합이 도왔/해쳤나)을 병행합니다. 본 연구의 기여는 대규모 벤치마크 수립이 아니라 특정 제약 환경에서의 통합·비교 검증이고, 질의 규모는 한계 절에 명시하겠습니다." (설계서 3-3, 3-4; 가이드 5-1, 9-2)

### Q4. proposed의 성능이 VLM 캡션 품질에 그대로 좌우되는 것 아닌가? 환각은 어떻게 통제하나?
**답변 초안:** "맞습니다. 캡션 품질 의존은 이 설계의 본질적 특성이고, 그래서 두 가지 공학적 통제를 넣었습니다. 첫째, 환각이 가장 잘 생기는 정적 구간 — 화면 변화 없이 음성만 있는 구간 — 은 프레임 차분으로 자동 판정해서, 정규화된 캡션 점수를 자막 점수로 치환합니다. 캡션을 버리는 게 아니라 치환하는 이유는 점수 스케일의 불연속을 막아 baseline과의 비교 공정성을 유지하기 위해서입니다. 둘째, 프롬프트에 '화면에 보이지 않는 것은 쓰지 마라'를 고정하고, 대표 프레임도 정보량이 많은 동적 프레임을 골라 캡션 재료 자체를 개선했습니다. 그리고 캡션이 나빠서 결합이 손해인 케이스는 실패가 아니라 결과입니다 — α grid search가 자막 쪽으로 수렴하면 그 자체가 '이 환경에서 캡션 기여는 제한적'이라는 정량 결론이 됩니다. 정적 치환 on/off ablation도 11주차에 계획돼 있습니다." (가이드 8-4, 2장; config caption_prompt)

### Q5. baseline이 너무 약한 것 아닌가? proposed에 유리하게 세팅된 비교 아닌가?
**답변 초안:** "그 우려 때문에 baseline을 '별도 구현'이 아니라 동일 파이프라인의 α=1.0 특수 경우로 규정했고, 별도 코드 경로를 금지했습니다. 즉 Whisper 자막, 임베딩 모델, per-query min-max 정규화, 정적 치환 인프라까지 전부 동일하고, 다른 것은 α 하나뿐입니다. 정적 치환은 α=1.0에서는 수학적으로 무영향이라 baseline에 불이익도 없습니다. 자막 단독 baseline 자체도 약체가 아닙니다 — Video Enriched RAG에서 ASR 단독이 시각 캡션 단독보다 높았고(HIT@1 0.741 vs 0.65), 저희 무발화 세그먼트도 빈 문자열 그대로 임베딩해 대칭성을 지켰습니다. 그리고 α는 dev에서만 선택해 test에 고정하므로 test에서 proposed에 유리한 α를 고르는 일은 구조적으로 불가능합니다." (가이드 8-4, 3장; 설계서 4-5, 3-1)

# 평가 규모 확장 계획 (B1) — 검정력 근거와 라벨링 프로토콜

목적: 현행 test n=39의 통계적 한계를 규모로 해소한다. **라벨링은 규율상 사용자 작업**
(프레임 실물 검증·검색 결과 참조 금지·유형 사전 등록) — 본 문서는 (a) 확장이 무엇을
사는지 정량화하고, (b) 규율 준수 프로토콜과 착수 스캐폴딩을 제공한다.

## 1. 왜 확장인가 — 검정력 실측 (docs 근거: results/eval_test.json per_query)

현재 test 39건의 관측 효과와, 유의(95% CI 0 배제)에 필요한 표본 추정:

| 지표 | 관측 Δ(proposed−base) | 현 95%CI 반폭 | 현 상태 | 유의 필요 n(근사) |
|---|---|---|---|---|
| MRR | +0.180 | ±0.126 | **이미 유의** | ~19 |
| hit@1 | (8-6: CI 0 배제) | — | **이미 유의** | — |
| hit@5 | +0.103 | ±0.140 | 0 포함(비유의) | **~73** |
| hit@10 | (8-6: CI 0 포함) | — | 0 포함 | (hit@5 이상) |

**결론: test를 39 → ~73(약 +34 질의, 이상적으로 신규 영상 2~3개에 분산)로 늘리면 hit@5가
유의로 전환될 가능성이 높다.** MRR·hit@1은 이미 유의라 규모 확장은 "이미 있는 결과의
신뢰도 승격"이지 새 주장 창출이 아니다 — ROI가 명확하다.

*한계(정직): 위 필요 n은 효과크기가 신규 영상에서도 유지된다는 정규근사 가정이다. 신규
영상에서 효과가 작아지면 더 필요할 수 있다(그 자체도 일반화의 정직한 검증이 된다).*

## 2. 라벨링 프로토콜 (절대규칙 준수 — 신규 라벨도 동일)

1. **프레임 실물 검증만.** GT는 캡션·자막 텍스트가 아니라 프레임을 직접 보고 판정한다
   (분석·라벨 공통 규율). 검색 결과를 보고 정답을 고르거나 고치지 않는다.
2. **유형 목표 사전 등록.** 병합 전에 신규분의 자막형/장면형/복합형 목표 수를 등록한다
   (현행 test 12/13/14 비율 참고, 콘텐츠 특성상 균등 강제는 아님 — 8-6).
3. **dev/test는 video_id 단위 분리.** 신규 영상은 test 전용으로만 넣고 dev와 겹치지 않게
   한다(누수 차단, M6 load_queries가 assert).
4. **gt_seg_idx 파생 규칙.** gt_start/end에서 1초 이상 겹치는 세그먼트 전부(없으면 최대
   겹침 1개) — derive_gt_seg_idx 계약. 다중 인스턴스 수동 추가는 wl_q03류 예외로만
   (새 예외 만들지 말 것, 8-6).
5. **queries.jsonl 스키마 그대로**(3-3): query_id/video_id/text/type/gt_start/gt_end/
   gt_seg_idx/split. 명세서(Excel)에서 라벨 후 export.

## 3. 착수 스캐폴딩 (실행 준비)

- **신규 test 영상 선정 기준:** 기존 4장르(여행/AI홍보/요리/테크)와 다른 장르 우선(예:
  브이로그 일상, 게임 실황, 강의/튜토리얼) — 일반화 폭을 넓힌다. 5~30분 한국어 영상.
- **인덱싱은 기존 파이프라인 그대로:** M1→M4 실행하면 신규 영상 인덱스 생성(변형 config
  불필요, 본 인덱스에 추가). 캡션 오염은 `--recaption-corrupted`로 자동 정리.
- **라벨 20~34건/영상**을 프레임 검증으로 작성 → queries.jsonl에 split="test"로 append →
  M6 재실행(확정 config). 이때가 유일하게 승인된 test 재평가(규모 확장 사유).
- **내가 할 수 있는 보조:** 신규 영상 인덱싱 실행, 프레임 추출·타임스탬프 후보 제시(정답
  판정은 사용자), 유형 사전등록 표 템플릿, 병합·무결성 검증(validate_gt_seg_idx) 실행.
  **정답 라벨 자체는 사용자가 프레임 검증으로 확정.**

## 4. 주의

- 이 확장은 **확정 config 불변** 전제. config가 바뀌는 게 아니라 평가 표본만 는다.
- 확장 후 재평가는 규율상 "확정 config가 바뀌는 예외"가 아니라 "평가 규모 확장"이므로,
  착수 전 사용자 승인 1회로 진행하고 접촉 이력(8-6)에 6회차로 기록한다.
